# Calls, voicemail, contacts & interactions

> **In one sentence:** Four small CoreData/SQLite stores — `CallHistory.storedata`, `AddressBook.sqlitedb`, `voicemail.db`, and `interactionC.db` — together reconstruct *who* the subject communicates with, *how often*, *which direction*, and *through which app*, and the last of them quietly logs the third-party-app contacts that the call and SMS logs never see.

## Why this matters

The previous lesson gave you the *content* of communications — the iMessage/SMS text in `sms.db`. This one gives you the **people graph** around it: the call log, the contact identities those numbers resolve to, the recorded voicemail audio, and — most valuable of all — the system's own aggregated *interaction* ledger that ranks every correspondent by frequency and recency across **every** communication app, not just Apple's. A defense of "I barely know that person" collapses against an `interactionC.db` row showing 412 WhatsApp interactions over six months with a first-contact date and a last-contact date. These stores are also where deleted-contact and deleted-call recovery lives, and where three different timestamp epochs sit side by side waiting to throw your timeline off by 31 years. Know each store's schema, its epoch, and what it captures that the others miss.

## Concepts

### The people graph: four stores, one network

iOS does not keep "communications" in one place. It keeps **identities**, **events**, **recordings**, and a **derived interaction graph** in four separate stores, each owned by a different daemon, each with its own schema and — critically — its own timestamp epoch:

```
                       ┌─────────────────────────────┐
   AddressBook.sqlitedb│  IDENTITY  (who is who)      │  ← contactsd / nameresolution
   ABPerson ───────────┤  name ↔ number ↔ email       │     epoch: Mac Absolute (2001)
                        └───────────┬─────────────────┘
                                    │ number/email is the join key
        ┌───────────────────────────┼───────────────────────────┐
        ▼                           ▼                           ▼
 ┌──────────────┐          ┌──────────────┐          ┌────────────────────┐
 │CallHistory   │ EVENTS   │ voicemail.db │ RECORDED │ interactionC.db    │ GRAPH
 │.storedata    │ (calls)  │  + *.amr     │ (audio)  │ (CoreDuet)         │ (frequency)
 │ ZCALLRECORD  │          │  voicemail   │          │ ZINTERACTIONS      │
 │ epoch: 2001  │          │ epoch: 1970! │          │ ZCONTACTS          │
 └──────────────┘          └──────────────┘          │ epoch: Mac Abs 2001│
   callservicesd             voicemail               └────────────────────┘
                                                        coreduetd  (device-only)
```

The number/email string is the universal join key. `AddressBook.sqlitedb` is the *resolver* (number → human name); the other three are *event sources*. Three of the four use **Mac Absolute Time** (CFAbsoluteTime, seconds since 2001-01-01 UTC); `voicemail.db`'s primary `date` column uses **Unix epoch** (seconds since 1970) — the single most common epoch mistake in this lesson. Hold that thought.

> 🖥️ **macOS contrast:** You met every one of these stores in macOS-mastery — `~/Library/Application Support/CallHistoryDB/CallHistory.storedata`, `~/Library/Application Support/AddressBook/`, and `~/Library/CoreDuet/People/interactionC.db`. The schemas are *byte-for-byte the same CoreData models*. The difference is provenance: on the Mac those stores are largely a **Continuity mirror** populated when "Calls on Other Devices" / Handoff push the iPhone's data over. On iOS **this is the primary copy** — the original system of record. When a Mac and an iPhone disagree, the iPhone store is authoritative and the Mac store is the downstream replica.

### CallHistory.storedata — the call log

**Path:** `/private/var/mobile/Library/CallHistoryDB/CallHistory.storedata` (+ `-wal`, `-shm` sidecars). In an iTunes/Finder backup it lands in the **WirelessDomain** under the hashed name `5a4935c78a5255723f707230a451d79c540d2741`.

Despite the `.storedata` extension it is an **ordinary SQLite database** generated by **CoreData** (the `callservicesd`/`CallHistory.framework` stack). CoreData stamps it with the usual bookkeeping tables — `Z_METADATA`, `Z_PRIMARYKEY`, `ZCALLDBPROPERTIES` — and one workhorse table, **`ZCALLRECORD`**, one row per call leg.

| Column | Meaning |
|---|---|
| `Z_PK` | CoreData primary key (the call's row id) |
| `ZADDRESS` | The remote party — phone number **or** Apple ID/email. **Stored as a BLOB**, not text |
| `ZDATE` | Call start, **Mac Absolute Time** (double) — add `978307200` for Unix |
| `ZDURATION` | Call length in **seconds** (double); `0` for a missed/declined call |
| `ZORIGINATED` | Direction: **`0` = incoming, `1` = outgoing** |
| `ZANSWERED` | `0` = not answered (missed/declined), `1` = answered |
| `ZCALLTYPE` | Media type: `1` = cellular/standard, `8` = FaceTime **video**, `16` = FaceTime **audio**. (The *app/transport* is **not** encoded here — it lives in `ZSERVICE_PROVIDER`. No documented "third-party-app" `ZCALLTYPE` value exists.) |
| `ZSERVICE_PROVIDER` | Bundle/UTI of the service that placed the call, e.g. `com.apple.Telephony` (cellular), `com.apple.FaceTime`, `net.whatsapp.WhatsApp`, `com.skype.skype` — this is how a third-party-app call is identified |
| `ZREAD` | Whether the (missed-call) entry was viewed |
| `ZCALL_CATEGORY` | Spam/identification category (Silence Unknown Callers, carrier spam flag) |
| `ZDISCONNECTED_CAUSE` | Why the call ended (provider-dependent) |
| `ZISO_COUNTRY_CODE` | Country inferred for the number |
| `ZUNIQUE_ID` | A GUID for the call leg |
| `ZFACE_TIME_DATA` | BLOB of FaceTime session metadata when applicable |

> 🔬 **Forensics note:** `ZADDRESS` is a **BLOB**, not a string. A naïve `SELECT ZADDRESS FROM ZCALLRECORD` returns raw bytes (or `<binary>` in the CLI); you must `CAST(ZADDRESS AS TEXT)` to read the number. Tools that "lose" call numbers almost always forgot this cast. The number is stored **as dialed/received** (often E.164 with `+` and country code, sometimes a bare local number), *not* normalized — so matching it back to `AddressBook` requires last-N-digit comparison, not string equality.

> 🔬 **Forensics note:** Several columns disambiguate a call that the UI shows identically. A missed FaceTime audio call is `ZORIGINATED=0, ZANSWERED=0, ZCALLTYPE=16, ZDURATION=0`. An outgoing WhatsApp call is `ZORIGINATED=1, ZSERVICE_PROVIDER='net.whatsapp.WhatsApp'` — the *app* is read off `ZSERVICE_PROVIDER`, while `ZCALLTYPE` still records only its audio/video media type (there is no "app" `ZCALLTYPE` code). Always report **direction (`ZORIGINATED`) + answered (`ZANSWERED`) + media type (`ZCALLTYPE`) + transport/app (`ZSERVICE_PROVIDER`)** together; any one alone is ambiguous.

**FaceTime and group calls** spread participants across a join: `ZCALLRECORD` rows link to **`Z_2REMOTEPARTICIPANTHANDLES`** (and per-participant handle rows) so a single group FaceTime enumerates each attendee's handle. If you only read `ZADDRESS` you see one party and miss the rest of a conference — check the participant join for any FaceTime/group leg.

> 🔬 **Forensics note — log depth & deletion:** The call log is **not** infinite. iOS keeps roughly the most recent ~1,000 entries (the exact cap is build-dependent — verify against your image), pruning oldest-first; "Recents" UI shows fewer. A user "Clear All Recents" deletes rows but the freed pages persist in the SQLite freelist and the `-wal` until checkpointed/vacuumed — carve them. Calls also sync via **iCloud ("Calls in iCloud")** when enabled, so a row's presence does not prove it originated on *this* handset; a cross-device merge can import calls placed elsewhere.

> 🔬 **Forensics note — anti-forensics & fabrication:** Because the store is plain SQLite, a jailbroken or backup-restore workflow can *insert* fabricated `ZCALLRECORD` rows or *edit* `ZDATE`. Tells: `Z_PK` values out of monotonic order versus `ZDATE`, a `ZDATE` that contradicts the device's own clock-change history, gaps in `ZUNIQUE_ID` allocation, or a `ZDATE` predating the contact's `AddressBook.CreationDate`. Cross-corroborate every load-bearing call against an independent store — `interactionC.db`, `knowledgeC`/Biome app-usage of `com.apple.mobilephone`, or carrier records (CDRs). Apple's clock-rollback behavior (and how it perturbs `CallHistory`) is documented in Cellebrite's "If I Could Turn Back Time."

### AddressBook.sqlitedb — the identity resolver

**Path:** `/private/var/mobile/Library/AddressBook/AddressBook.sqlitedb` (contact photos live separately in `AddressBookImages.sqlitedb`). Backup name (HomeDomain): `31bb7ba8914766d4ba40d6dfb6113c8b614be442`.

This is the store that turns `+15125550143` into "Jordan Reyes." Its quirk is that one contact's data is **spread across three tables** in a property/multi-value model — a contact with three phone numbers and two emails is *one* `ABPerson` row and *five* `ABMultiValue` rows:

```
ABPerson (ROWID=42)  First="Jordan" Last="Reyes" Organization="Acme"
   │                  CreationDate / ModificationDate  (Mac Absolute)
   │  record_id ─────────────────────────────┐
   ▼                                          ▼
ABMultiValue                          ABMultiValueLabel
 record_id=42 property=3 value="+15125550143" label=1 ──► value="_$!<Mobile>!$_"
 record_id=42 property=3 value="+15125550199" label=2 ──► value="_$!<Work>!$_"
 record_id=42 property=4 value="jr@acme.com"  label=3 ──► value="_$!<Home>!$_"
            property: 3=phone  4=email  5=address  22=URL  13=birthday/date ...
```

- **`ABPerson`** — one row per contact: `ROWID`, `First`, `Last`, `Middle`, `Organization`, `Department`, `JobTitle`, `Nickname`, `Note`, `Birthday`, and the two forensic gold columns **`CreationDate`** and **`ModificationDate`** (both Mac Absolute, floating-point).
- **`ABMultiValue`** — the repeating values: `record_id` (→ `ABPerson.ROWID`), `property` (type code: 3=phone, 4=email…), `value` (the actual number/email), `label` (→ `ABMultiValueLabel.ROWID`).
- **`ABMultiValueLabel`** — decodes the label id to a string like `_$!<Mobile>!$_`, `_$!<Home>!$_`, `_$!<Work>!$_` (Apple's bracketed-token convention).

Supporting tables matter too: **`ABPhoneLastFour`** holds the *normalized* last-four digits Apple uses for fast caller matching (your shortcut for joining numbers across stores), **`ABRecent`** caches recently-used addresses, **`ABGroup`**/**`ABGroupMembers`** model contact groups, **`ABStore`** records the backing account (local vs. iCloud vs. Exchange/CardDAV) each contact came from, and **`ABPersonChanges`** is a change journal for iCloud sync.

The `property` integer in `ABMultiValue` is the type discriminator you must decode to know what a value *is*:

| `property` | Value type | | `property` | Value type |
|---|---|---|---|---|
| `3` | phone number | | `13` | birthday / related date |
| `4` | email address | | `22` | URL / homepage |
| `5` | postal address | | `23` | related name (spouse, child…) |
| `12` | IM / social handle | | `46` | social profile (Twitter/X, etc.) |

(Codes are stable across iOS versions but **verify against `ABMultiValueLabel` on your image** — Apple has added handle types over time.) Note `AddressBook.sqlitedb` is a *hand-rolled* `AB`-prefixed schema, **not** CoreData — there are no `Z`-prefixed tables here. The `ABPerson.StoreID` → `ABStore` link matters forensically: a contact sourced from a corporate Exchange/CardDAV account behaves and syncs differently from a locally-created one, and "local-only" contacts are the ones a subject deliberately kept off the cloud.

> 🔬 **Forensics note:** `ABPerson.CreationDate` and `ModificationDate` answer "when did the subject *add or edit* this contact?" — often more probative than the contact's content. A burner number saved minutes before a key call, or a contact whose `ModificationDate` clusters with an incident, is a lead. These are **device-local edit times**, distinct from the call/message events themselves. (Both are Mac Absolute — `+978307200`.)

> 🔬 **Forensics note:** The display name your tool shows next to a call is sometimes a **cached/denormalized** copy living in `CallHistory` or `interactionC`, not a live `AddressBook` lookup. If a contact was renamed or deleted after a call, the call log may still carry the *old* cached name while `AddressBook` shows the new one (or nothing). Report the raw number and the resolved name as separate facts; note the source of each.

### voicemail.db — recorded evidence on disk

**Path:** `/private/var/mobile/Library/Voicemail/voicemail.db`, with the audio files alongside as `<ROWID>.amr` (AMR-NB codec) in the same `Voicemail/` directory.

Unlike the other three stores this is a **plain hand-rolled SQLite** table (no CoreData `Z`-prefix), and it is the **epoch outlier**:

| Column | Meaning |
|---|---|
| `ROWID` | Primary key — **and the audio filename**: `ROWID` 107 → `107.amr` |
| `remote_uid` | Internal caller id reference |
| `date` | When the voicemail was left — **Unix epoch (seconds since 1970)**, NOT Mac Absolute |
| `sender` | Caller's phone number (`NULL` if withheld/blocked) |
| `callback_num` | Callback number (may differ from `sender`) |
| `duration` | Length in seconds |
| `expiration` | When the carrier/device will auto-expire it (Unix epoch) |
| `trashed_date` | When moved to "Deleted" — **Mac Absolute (2001)!** |
| `flags` | Bitmask: read/unread, deleted, etc. |

> 🔬 **Forensics note — the two-epoch trap in ONE table:** `voicemail.date` is **Unix (1970)** but `voicemail.trashed_date` is **Mac Absolute (2001)** in the *same row*. Apply `+978307200` to `trashed_date` and you get a date ~31 years in the future; apply it to `date` and you get a date ~31 years in the past. This is the canonical worked example of "the iOS timestamp zoo" — convert each column by its own epoch, never globally. (See [[00-the-ios-timestamp-zoo]].)

> 🔬 **Forensics note — carving deleted voicemail:** A deleted voicemail is *soft-deleted* — its row gets a `trashed_date` and a deleted `flags` bit but the **`<ROWID>.amr` file often remains on disk** until a later purge. So even when the row is gone you may still have the audio, and even when the audio is gone the row's metadata (caller, date, duration) may survive in the DB or its `-wal`. Recover both halves independently.

**Transcription & Live Voicemail.** Since iOS 10, voicemails carry **transcripts** (best-effort speech-to-text); the transcript text is stored with the voicemail (DB column / sidecar — *verify exact location on your image*). iOS 17 added **Live Voicemail** (on-device, real-time Neural-Engine transcription as the caller speaks), and **iOS 26 Call Screening** is built on the same pipeline — unknown callers are auto-answered, prompted to state their purpose, and live-transcribed before the phone ever rings. These features expand the *recorded/transcribed* surface; treat the exact iOS 26 call-screening store path as **research-at-author-time** rather than assuming `voicemail.db`.

**Provenance details worth knowing.** The audio is **AMR-NB** (Adaptive Multi-Rate narrowband, the carrier-grade telephony codec) — a `file 107.amr` shows the `#!AMR\n` magic, and the codec/bitrate are themselves a weak authenticity check (a re-encoded or spliced file often won't match AMR-NB framing). The same directory holds the user's outgoing **greeting** recording and a `directory`/token file tying the device to the carrier's **Visual Voicemail** account; that token, plus `expiration` timestamps, is provenance that the voicemail came through the carrier's VVM service rather than being side-loaded. Greetings and audio are *user/third-party content*, so they carry the same authorization weight as the messages themselves.

> ⚖️ **Authorization:** Voicemail `.amr` files are **recorded audio of third parties** who did not consent to your examination. Depending on jurisdiction, the *content* of recorded communications can fall under stricter authority (wiretap/stored-communications statutes) than the metadata. Confirm your warrant/authorization scope covers recorded audio content, log every file you extract, and hash the `.amr` set into your chain of custody before you transcode or play anything.

### interactionC.db — the interaction graph (the crown jewel)

**Path:** `/private/var/mobile/Library/CoreDuet/People/interactionC.db` (+ `-wal`/`-shm`). Owner: **`coreduetd`**, the CoreDuet "duet" intelligence daemon that feeds Siri Suggestions, Proactive, and the share sheet's people row.

This is the store that earns this lesson. `CallHistory` knows phone calls; `sms.db` knows iMessage/SMS; **`interactionC.db` knows *everything else too*** — it records an "interaction" whenever the user communicates through *any* app that adopts the Intents/CoreDuet donation API: WhatsApp messages, Instagram DMs, Gmail, Outlook, Telegram, Signal, third-party VoIP — with **direction, app, and per-contact running totals**. The call/SMS logs are blind to all of that; CoreDuet is not.

Two tables carry the weight:

**`ZINTERACTIONS`** — one row per interaction event:

| Column | Meaning |
|---|---|
| `ZSTARTDATE` / `ZENDDATE` | Event window (Mac Absolute, `+978307200`) |
| `ZCREATIONDATE` | When the row was donated |
| `ZBUNDLEID` | The app the user was in — `com.apple.mobilephone`, `com.apple.MobileSMS`, `com.apple.mobilemail`, `net.whatsapp.WhatsApp`, `com.burbn.instagram`, `org.whispersystems.signal`, … |
| `ZTARGETBUNDLEID` | Target app, when the interaction routed elsewhere |
| `ZDIRECTION` | **`0` = incoming, `1` = outgoing** |
| `ZSENDER` | FK → `ZCONTACTS.Z_PK` (the other party) |
| `ZRECIPIENTCOUNT` | How many recipients (1 = DM, >1 = group) |
| `ZISRESPONSE` | Whether this was a reply |
| `ZDOMAINIDENTIFIER` | Logical channel (e.g. messaging vs. calling) |
| `ZGROUPNAME` / `ZCONTENTURL` / `ZDERIVEDINTENTIDENTIFIER` | Group thread, attached content, inferred Siri intent |

**`ZCONTACTS`** — one row per distinct correspondent, with **aggregated lifetime counters** — this is the frequency/recency engine:

| Column | Meaning |
|---|---|
| `ZDISPLAYNAME` | Cached human name |
| `ZIDENTIFIER` | The number/email that keys the contact |
| `ZPERSONID` | Link back to the AddressBook person |
| `ZINCOMINGSENDERCOUNT` | Total messages **received from** this contact |
| `ZOUTGOINGRECIPIENTCOUNT` | Total messages **sent to** this contact |
| `ZINCOMINGRECIPIENTCOUNT` | Group-incoming tally |
| `ZFIRSTINCOMINGSENDERDATE` / `ZLASTINCOMINGSENDERDATE` | **First and last** time they contacted the subject |
| `ZFIRSTOUTGOINGRECIPIENTDATE` / `ZLASTOUTGOINGRECIPIENTDATE` | First/last time the subject contacted them |
| `ZCREATIONDATE` | When CoreDuet first saw this contact |

Attachments hang off interactions through the auto-generated CoreData join table `Z_1INTERACTIONS` (→ `ZATTACHMENT`); the per-contact link is `ZINTERACTIONS.ZSENDER` → `ZCONTACTS.Z_PK`, and CoreData auto-generates a further many-to-many recipient join table whose exact name is **build-dependent** — enumerate it with `.tables` rather than hard-coding it. Sarah Edwards' **APOLLO** `interaction_contact_interactions` module stitches these joins into a flat timeline so you don't hand-write them.

**How interactions get there (the donation mechanism).** CoreDuet does not snoop packets — apps *donate* interactions to it through public API. Apple's own Phone/Messages/Mail donate automatically; third-party apps opt in by calling `INInteraction.donate(completion:)` (Intents framework) or by publishing an `NSUserActivity`/`CSSearchableItem` when the user messages or calls a contact. The donation carries an `INPerson` (the handle), an intent type (`INSendMessageIntent`, `INStartCallIntent`…), and a direction, which `coreduetd` lands as a `ZINTERACTIONS` row and folds into the `ZCONTACTS` aggregate counters. This is *why* a privacy app can still leak its contact graph here: the conversation body is end-to-end encrypted, but the app voluntarily told the OS "the user just messaged this person" so Siri can suggest them. It also means **coverage is per-app and best-effort** — an app that never adopts donation is invisible to `interactionC.db`, so absence of a contact is not proof of no contact.

> 🖥️ **macOS contrast:** The donation API is identical on macOS — `INInteraction`/`NSUserActivity` feed the same `coreduetd` and the same `interactionC.db` schema. On the Mac you mostly saw Apple-app donations (Mail, Messages, Calls-via-Continuity); on iOS the donor pool is the whole installed third-party app set, which is exactly what makes the iOS copy richer than the one you parsed in macOS-mastery.

> 🔬 **Forensics note — relationship strength on a plate:** Because `ZCONTACTS` stores *aggregate counters plus first/last dates per contact*, you can rank every correspondent by total volume and recency with a single query — no per-message counting needed. `ORDER BY (ZINCOMINGSENDERCOUNT + ZOUTGOINGRECIPIENTCOUNT) DESC` is an instant "closest contacts" list, and `ZFIRSTINCOMINGSENDERDATE` answers "when did this relationship begin?" This is the artifact that converts "do you know this person?" into a number. **Caveat:** the *exact* accounting of the count columns is imperfectly documented (Sarah Edwards' CoreDuet write-up flags them as not fully pinned down — group vs. 1:1 weighting in particular). Treat the *relative ranking* and the first/last dates as robust; validate any load-bearing *absolute* count against the underlying `ZINTERACTIONS` rows for that contact.

> 🔬 **Forensics note — it catches what the others miss:** A subject who does all their sensitive comms in Signal or WhatsApp leaves *nothing* in `CallHistory`/`sms.db` for that traffic — but every donated interaction still lands in `interactionC.db` with `ZBUNDLEID='org.whispersystems.signal'` and a contact, direction, and timestamp. The **content** is encrypted and out of reach; the **metadata of contact** is right here. CoreDuet is frequently the *only* on-device evidence that a third-party-app conversation happened at all.

> 🔬 **Forensics note — corroborate with Biome/knowledgeC:** CoreDuet is the same intelligence family that feeds `knowledgeC` and the newer **Biome/SEGB** streams, and from iOS 17 onward Apple has been shifting pattern-of-life telemetry toward Biome (see [[02-biome-and-segb-streams]]). Practically, an `interactionC` interaction with `com.apple.MobileSMS` should line up with a `/app/inFocus` or `/app/usage` window for Messages in `knowledgeC`/Biome at the same minute. When they *don't* align — an interaction with no corresponding app-usage, or vice versa — you have either a donation from a background/extension context or a tampering indicator worth chasing. Treat the three stores as mutually corroborating, not redundant.

> 🖥️ **macOS contrast:** This is the exact same `interactionC.db` you parsed on macOS at `~/Library/CoreDuet/People/`. On the Mac it tracks Mail/Messages/Calls interactions for Siri Suggestions; on iOS the same daemon (`coreduetd`) and schema track the full mobile app surface. If you ran APOLLO's CoreDuet modules in macOS-mastery, the modules apply unchanged here — only the device of origin differs.

### Deleted-data recovery across the four stores

None of these stores hard-delete on the user's first action — each leaves a recoverable trail, but a *different* one, so know which lever to pull per store:

```
CallHistory.storedata   "Clear All Recents" → rows unlinked, pages → SQLite freelist
                         + newest calls may sit only in -wal until checkpoint
                         → recover via freelist/unallocated-page carving + -wal replay
AddressBook.sqlitedb    contact deleted → ABPerson row removed, but ABPersonChanges
                         journals the change, and an iCloud "Recently Deleted" copy and
                         backup snapshots may retain it → diff against an earlier backup
voicemail.db            soft-delete → trashed_date set + flags bit; <ROWID>.amr usually
                         STILL ON DISK → recover audio independently of the row
interactionC.db         aggregate counters in ZCONTACTS persist even after individual
                         ZINTERACTIONS rows age out → totals/first-last dates survive the
                         detail; deleted detail rows carve from freelist/-wal
```

The general DFIR discipline (covered in [[14-deleted-data-recovery]]) applies: **acquire the `-wal`/`-shm`, then carve freelist and unallocated pages** before you trust a "0 results" query. The standout here is voicemail — the database row and the `.amr` audio are *two separate deletable objects*, and losing one does not mean losing the other.

### The epoch zoo for this lesson

Four stores, three epochs. Memorize the per-column mapping; do not apply one offset globally.

| Store · column | Epoch | Convert to Unix |
|---|---|---|
| `CallHistory` · `ZDATE` | Mac Absolute (2001) | `+ 978307200` |
| `AddressBook` · `CreationDate` / `ModificationDate` / `Birthday` | Mac Absolute (2001) | `+ 978307200` |
| `voicemail` · `date` / `expiration` | **Unix (1970)** | **none** |
| `voicemail` · `trashed_date` | Mac Absolute (2001) | `+ 978307200` |
| `interactionC` · all `Z*DATE` columns | Mac Absolute (2001) | `+ 978307200` |

### Putting it together: one number across four stores

The stores are weak alone and strong together. Take a single unknown number, `+15125550143`, and walk it through all four — this is the model the unified-timeline lesson generalizes:

```
1. AddressBook   → ABMultiValue.value last-4 = 0143 → ABPerson "Jordan Reyes",
                   CreationDate 2026-03-02 09:14  →  saved the morning of incident day
2. CallHistory   → 3× ZCALLRECORD: 1 outgoing answered (ZORIGINATED=1,ZANSWERED=1,
                   ZCALLTYPE=1, 214s) + 2 missed incoming, all 2026-03-02 afternoon
3. voicemail.db  → 1 row, date (UNIX) 2026-03-02 16:41, duration 38s, 38.amr on disk
                   → transcript + audio of what was actually said
4. interactionC  → ZCONTACTS row for 0143: total 96 interactions, ZFIRSTINCOMINGSENDERDATE
                   2025-11-08; its linked ZINTERACTIONS rows split by ZBUNDLEID across
                   com.apple.mobilephone AND net.whatsapp.WhatsApp (mostly WhatsApp)
                   →  relationship is FOUR MONTHS OLD, not new
```

Each store answered a different question — identity + when-saved (AddressBook), the call events (CallHistory), the recorded words (voicemail), and the *true depth and channel* of the relationship including the WhatsApp traffic invisible to every other store (interactionC). The `interactionC` first-contact date directly rebuts a "we'd only just met" claim that the call log alone would have seemed to support. **No single store tells this story; the join does.**

## Hands-on

There is no on-device shell — everything runs on the Mac against a **copy** of each store. CoreData/SQLite write-lock even on `SELECT` and create `-wal`/`-shm`; copy the database **and both sidecars** before you touch them, and prefer opening read-only.

```bash
# Forensic-safe open of any of these stores (read-only URI; never the live file)
cp CallHistory.storedata CallHistory.storedata-wal CallHistory.storedata-shm /tmp/ch/ 2>/dev/null
sqlite3 'file:/tmp/ch/CallHistory.storedata?mode=ro&immutable=1' '.tables'
```

**Call log — direction/answered/channel, with the BLOB cast:**

```bash
sqlite3 'file:/tmp/ch/CallHistory.storedata?mode=ro&immutable=1' "
SELECT
  datetime(ZDATE + 978307200, 'unixepoch', 'localtime')      AS start,
  CAST(ZADDRESS AS TEXT)                                      AS number,
  CASE ZORIGINATED WHEN 0 THEN 'IN' WHEN 1 THEN 'OUT' END     AS dir,
  CASE ZANSWERED  WHEN 1 THEN 'answered' ELSE 'missed' END    AS ans,
  CASE ZCALLTYPE
       WHEN 1 THEN 'cellular/std' WHEN 8 THEN 'FT-video'
       WHEN 16 THEN 'FT-audio' ELSE ZCALLTYPE END             AS media,
  ZSERVICE_PROVIDER                                           AS via,   -- app/transport: com.apple.Telephony, net.whatsapp.WhatsApp …
  CAST(ZDURATION AS INT)                                      AS secs
FROM ZCALLRECORD
ORDER BY ZDATE DESC LIMIT 25;"
```

**Contact resolution — the three-table join:**

```bash
sqlite3 'file:/tmp/ab/AddressBook.sqlitedb?mode=ro&immutable=1' "
SELECT p.ROWID,
       p.First || ' ' || IFNULL(p.Last,'')                      AS name,
       l.value                                                  AS label,
       v.value                                                  AS number_or_email,
       datetime(p.CreationDate     + 978307200,'unixepoch')     AS created,
       datetime(p.ModificationDate + 978307200,'unixepoch')     AS modified
FROM ABPerson p
JOIN ABMultiValue v       ON v.record_id = p.ROWID
LEFT JOIN ABMultiValueLabel l ON l.ROWID = v.label
ORDER BY p.ModificationDate DESC LIMIT 40;"
```

**Voicemail — note each column's own epoch:**

```bash
sqlite3 'file:/tmp/vm/voicemail.db?mode=ro&immutable=1' "
SELECT ROWID,
       datetime(date,'unixepoch','localtime')                  AS left_at,   -- UNIX
       sender, callback_num, duration,
       CASE WHEN trashed_date>0
            THEN datetime(trashed_date+978307200,'unixepoch')               -- 2001
            ELSE NULL END                                       AS trashed,
       flags
FROM voicemail ORDER BY date DESC LIMIT 25;"
# the audio for ROWID 107 is ./107.amr
afinfo 107.amr            # confirm AMR-NB codec / sample rate before relying on it
shasum -a 256 107.amr     # hash into chain of custody BEFORE any transcode
afconvert 107.amr 107.wav -d LEI16   # AMR-NB → 16-bit PCM WAV to play/transcribe
# afconvert AMR (samr) decode is macOS-build-dependent; if it fails, fall back to:
#   ffmpeg -i 107.amr 107.wav
```

**Interaction graph — rank correspondents by volume & recency (the money query):**

```bash
sqlite3 'file:/tmp/ic/interactionC.db?mode=ro&immutable=1' "
SELECT ZDISPLAYNAME, ZIDENTIFIER,
       ZINCOMINGSENDERCOUNT   AS rcvd,
       ZOUTGOINGRECIPIENTCOUNT AS sent,
       (ZINCOMINGSENDERCOUNT + ZOUTGOINGRECIPIENTCOUNT) AS total,
       datetime(ZFIRSTINCOMINGSENDERDATE+978307200,'unixepoch') AS first_in,
       datetime(ZLASTINCOMINGSENDERDATE +978307200,'unixepoch') AS last_in
FROM ZCONTACTS
ORDER BY total DESC LIMIT 20;"

# per-app interaction breakdown (catches WhatsApp/Signal/etc.)
sqlite3 'file:/tmp/ic/interactionC.db?mode=ro&immutable=1' "
SELECT ZBUNDLEID, COUNT(*) n,
       SUM(ZDIRECTION=0) incoming, SUM(ZDIRECTION=1) outgoing
FROM ZINTERACTIONS GROUP BY ZBUNDLEID ORDER BY n DESC;"
```

**Getting the files off (Mac-side acquisition).** From a backup or a logical/full-FS image:

> ⚠️ **ADVANCED:** Pulling these stores from a *live* device (`pymobiledevice3 backup2`, `idevicebackup2`) requires the device unlocked and paired, and writes a fresh pairing record + a backup that can perturb on-device state; an **encrypted** backup is mandatory to capture `CallHistory`/voicemail/contacts at full fidelity (an unencrypted backup omits some of them). Never run acquisition against the only copy of evidence — image first, hash, then operate on the copy. A full-file-system extraction (checkm8/usbliter8 on ≤A13, or an agent on a supported build) gives you the live DBs *plus* their `-wal`/`-shm` and the `.amr` set that a backup may not fully include. ## 🧪 Labs

> Substrate honesty up front: **`interactionC.db`, `CallHistory.storedata`, and `voicemail.db` are populated by device-only daemons (`coreduetd`, `callservicesd`) that DO NOT run in the iOS Simulator.** The Simulator can teach you `AddressBook` structure (Contacts.app works) but will not produce a call log, voicemail audio, or a CoreDuet interaction graph. Use a **public sample image** for those three.

### Lab 1 — AddressBook schema on the Simulator *(substrate: Xcode Simulator; fidelity: schema-faithful, but no `coreduetd`/call data, and Simulator stores sit unencrypted with no Data Protection)*

1. Boot a sim and add 2–3 contacts (multiple numbers + an email each) in Contacts.app:
   `xcrun simctl boot "iPhone 17 Pro"; open -a Simulator`
2. Locate the container on the Mac:
   `find ~/Library/Developer/CoreSimulator/Devices -name AddressBook.sqlitedb 2>/dev/null`
3. Copy it (+ any `-wal`/`-shm`) and run the three-table join query from Hands-on.
4. Edit one contact in the UI, re-copy, and diff `ModificationDate` — confirm only the touched person's row advanced. This proves what `ModificationDate` does and does not capture.

> 🖥️ **macOS contrast (use your own Mac as a substrate):** With no iOS device, your **own Mac is a real, populated copy** of three of these four stores — `~/Library/Application Support/CallHistoryDB/CallHistory.storedata`, `~/Library/Application Support/AddressBook/`, and `~/Library/CoreDuet/People/interactionC.db`. The CoreData schemas are identical to iOS, so you can practice the exact `ZCALLRECORD`/`ZINTERACTIONS` queries on live, non-synthetic data (copy-before-query still applies). The only iOS-exclusive store here is `voicemail.db` (no Mac equivalent). This is the highest-fidelity device-free substrate available for everything except the voicemail audio.

### Lab 2 — Call-log reconstruction from a sample image *(substrate: public sample forensic image, e.g. Josh Hickman's iOS reference image; fidelity: real device data, real epochs)*

1. Extract `CallHistory.storedata` (+ sidecars) from the image's `WirelessDomain` path.
2. Run the call-log query. **Deliberately omit the `CAST(ZADDRESS AS TEXT)`** first and observe the blob bytes — then add the cast and watch the numbers appear. Internalize the BLOB gotcha.
3. Find every FaceTime leg (`ZCALLTYPE IN (8,16)`) and, for any group leg, follow `Z_2REMOTEPARTICIPANTHANDLES` to enumerate participants.
4. Identify third-party-app calls by a non-Apple `ZSERVICE_PROVIDER` (e.g. `net.whatsapp.WhatsApp`) — the app is named *there*, not by any `ZCALLTYPE` value — and confirm `ZCALLTYPE` still only encodes the audio/video media type.

### Lab 3 — Frequency/recency ranking from `interactionC.db` *(substrate: public sample image; fidelity: device-only daemon, Simulator cannot produce this)*

1. Pull `interactionC.db` (+ `-wal`/`-shm`) from the sample image.
2. Run the "rank correspondents by volume & recency" query. Who are the top 5? When did each relationship begin (`ZFIRSTINCOMINGSENDERDATE`)?
3. Run the per-`ZBUNDLEID` breakdown. Identify at least one **third-party app** with interactions and confirm those same conversations are **absent** from `CallHistory`/`sms.db` — the core teaching point.
4. Cross-check: pick a top contact's `ZIDENTIFIER`, last-4-match it against `ABPhoneLastFour` to resolve the human name.

### Lab 4 — The voicemail epoch trap + audio carving *(substrate: public sample image / read-only walkthrough; fidelity: real epochs, real `.amr`)*

1. Query `voicemail` with the **wrong** epoch on purpose: apply `+978307200` to `date`. Note the ~2052 timestamp — feel the bug. Then convert `date` as plain Unix and `trashed_date` as Mac Absolute. Two epochs, one table.
2. For a row with `trashed_date > 0`, check whether `<ROWID>.amr` still exists on disk despite the soft-delete.
3. Transcode one `.amr` to WAV with `afconvert` and confirm playback. Hash the original `.amr` first and record it.

### Lab 5 — One unified people timeline *(substrate: public sample image; tooling: iLEAPP + APOLLO)*

1. Run iLEAPP across the image; open the Call History, Contacts, Voicemail, and InteractionC reports.
2. Run APOLLO's CoreDuet modules for the interaction timeline.
3. Merge into one CSV keyed on number (last-4 normalized) and sorted by time. For one target contact, assemble: contact creation date → first interaction → call legs → voicemails → last interaction. You have just built a single correspondent's relationship history from four stores — the deliverable a report actually needs. (Carry this into [[01-building-a-unified-timeline]].)
4. **Validation pass:** pick one `interactionC` row with `ZBUNDLEID='com.apple.mobilephone'` and confirm a matching `ZCALLRECORD` exists at the same time. Then find an `interactionC` row whose `ZBUNDLEID` is a third-party app and confirm it has **no** counterpart in `CallHistory`/`sms.db`. Documenting both the corroboration and the gap is what separates a defensible report from an assertion.

## Pitfalls & gotchas

- **`voicemail.date` is Unix, not Mac Absolute.** The #1 timeline error in this lesson. And `trashed_date` in the *same table* IS Mac Absolute. Convert per column.
- **`ZADDRESS` is a BLOB.** No `CAST(... AS TEXT)` = no numbers (you'll see `<binary>`/empty). True anywhere a CoreData store holds an identifier as a blob.
- **Copy the `-wal`/`-shm` too.** Recent calls, contacts, voicemails, and interactions may live *only* in the write-ahead log until checkpoint. Copying the bare `.db` silently drops the newest evidence and can leave you with a stale snapshot.
- **Numbers aren't normalized consistently.** `CallHistory.ZADDRESS` stores the number as dialed; `AddressBook` stores it as typed; `interactionC.ZIDENTIFIER` varies. Join on **last-N digits** (`ABPhoneLastFour` is your friend), never string equality, or you'll under-count a contact.
- **Cached names lie.** A name shown beside a call may be a stale denormalized cache from before a rename/delete. Treat raw number and resolved name as separate, separately-sourced facts.
- **The Simulator won't help for three of the four stores.** `coreduetd`/`callservicesd` don't run there; only `AddressBook` is meaningfully reproducible on the Simulator. Don't conclude "no calls/interactions" from a Simulator container.
- **iCloud sync = cross-device contamination.** "Calls in iCloud," iCloud Contacts, and CoreDuet's iCloud syncing mean a row can originate on *another* of the subject's devices. Presence ≠ origination. Note sync state from the account configuration before attributing an event to this handset. And with **Advanced Data Protection** enabled, the iCloud-side copies of these stores are end-to-end encrypted — cloud acquisition won't hand them to you (see [[06-icloud-acquisition-and-advanced-data-protection]]).
- **`.storedata` is just SQLite.** Don't be thrown by the extension — open `CallHistory.storedata` directly with `sqlite3`. Conversely, don't assume a `.db` is hand-rolled: `interactionC.db` is CoreData (`Z`-prefixed) while `voicemail.db` is not.
- **Deletion is soft, then lazy.** Cleared call rows persist in freelist/WAL; trashed voicemails keep their `.amr`; `ABPersonChanges` journals contact edits. Carve before you conclude "deleted."
- **An unencrypted backup is a lossy backup.** iTunes/Finder backups omit several of these stores (and Keychain) unless the backup is **encrypted**. If you acquired an unencrypted backup and the call log/voicemail/contacts look thin or empty, that's the acquisition method, not the device — re-acquire encrypted or go full-FS before concluding the data isn't there.
- **`property` codes and label tokens drift.** The `ABMultiValue.property` integers and `_$!<…>!$_` label tokens are stable but Apple periodically adds handle types (new social networks, RCS identifiers). Decode against the device's own `ABMultiValueLabel` rather than a hard-coded table from an older iOS.
- **iOS 26 surfaces shift.** Call Screening / Live Voicemail and Phone-app redesigns move and add data; don't assume a 26.x device matches a 17.x layout. Verify paths on the actual image; treat the call-screening store location as research-at-author-time.

## Key takeaways

- The **people graph** is four stores: `AddressBook` (identity), `CallHistory.storedata` (call events), `voicemail.db` + `.amr` (recorded audio), and `interactionC.db` (the cross-app interaction graph). The number/email is the universal join key.
- **`interactionC.db` is the crown jewel** — it logs interactions through *every* communication app (WhatsApp, Signal, Instagram, mail…) with direction, app bundle id, and per-contact lifetime counts + first/last dates. It is often the only on-device evidence a third-party-app conversation occurred.
- Report calls as **direction (`ZORIGINATED`) + answered (`ZANSWERED`) + channel (`ZCALLTYPE`/`ZSERVICE_PROVIDER`)** together; any one alone is ambiguous. Remember `ZADDRESS` is a BLOB.
- **Three epochs across four stores:** Mac Absolute for CallHistory/AddressBook/interactionC, **Unix** for `voicemail.date`, and Mac Absolute again for `voicemail.trashed_date` in the same row.
- `ABPerson.CreationDate`/`ModificationDate` time **when a contact was added/edited** — frequently more probative than the contact's content.
- **Copy DB + `-wal` + `-shm`, open read-only, join on last-N digits.** The newest evidence often lives only in the WAL; numbers aren't normalized for string equality.
- `interactionC` coverage is **donation-based and per-app** (`INInteraction`/`NSUserActivity` → `coreduetd`): rich for apps that adopt it, blind to apps that don't — so it sees more than the call/SMS logs, but absence still isn't proof of no contact. Corroborate load-bearing events across stores (`interactionC` ↔ `CallHistory` ↔ `knowledgeC`/Biome app-usage); mismatches flag background/extension donations or tampering.
- On iOS these stores are the **primary** copy; on the Mac you knew them as a **Continuity mirror**. iCloud sync and ADP change what's available cross-device and from the cloud. The Simulator only reproduces `AddressBook`, so use a public sample image for call/voicemail/interaction data — and note your own Mac is a high-fidelity, device-free substrate for three of the four.

## Terms introduced

| Term | Definition |
|---|---|
| `CallHistory.storedata` | CoreData/SQLite call log at `/private/var/mobile/Library/CallHistoryDB/`; main table `ZCALLRECORD` |
| `ZCALLRECORD` | One row per call leg: `ZADDRESS`, `ZDATE`, `ZDURATION`, `ZORIGINATED`, `ZANSWERED`, `ZCALLTYPE`, `ZSERVICE_PROVIDER` |
| `ZORIGINATED` | Call direction flag in `ZCALLRECORD`: 0 = incoming, 1 = outgoing |
| `ZCALLTYPE` | Call media type: 1 = cellular/standard, 8 = FaceTime video, 16 = FaceTime audio (the app/transport is in `ZSERVICE_PROVIDER`, not here) |
| `Z_2REMOTEPARTICIPANTHANDLES` | Join table enumerating participants of a FaceTime/group call leg |
| `AddressBook.sqlitedb` | Contacts store at `/private/var/mobile/Library/AddressBook/`; the number↔name resolver |
| `ABPerson` | One row per contact (name, org, `CreationDate`, `ModificationDate`) |
| `ABMultiValue` / `ABMultiValueLabel` | Repeating phone/email/URL values, and their human-readable labels (`_$!<Mobile>!$_`) |
| `ABPhoneLastFour` | Normalized last-four-digit table used for fast caller matching / cross-store joins |
| `voicemail.db` | Plain SQLite voicemail metadata; `date` is **Unix epoch**, `trashed_date` is Mac Absolute |
| `.amr` | AMR-NB audio file for a voicemail, named `<ROWID>.amr` matching `voicemail.ROWID` |
| Live Voicemail / Call Screening | iOS 17+/26 on-device real-time voicemail transcription and unknown-caller screening |
| `interactionC.db` | CoreDuet contact-interaction graph at `/private/var/mobile/Library/CoreDuet/People/` |
| `coreduetd` | CoreDuet daemon that donates interactions to `interactionC.db` (device-only; not in Simulator) |
| `ZINTERACTIONS` | One row per interaction: `ZBUNDLEID`, `ZDIRECTION`, `ZSTARTDATE`, `ZSENDER` |
| `ZCONTACTS` | Per-correspondent aggregates: incoming/outgoing counts, first/last contact dates |
| Mac Absolute Time | CFAbsoluteTime epoch (2001-01-01 UTC); add 978307200 to convert to Unix |
