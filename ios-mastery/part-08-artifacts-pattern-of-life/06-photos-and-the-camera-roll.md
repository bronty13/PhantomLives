# Photos & the camera roll

> **In one sentence:** `Photos.sqlite` is a Core Data catalog that ties every image and video on the device to a *who* (on-device face clustering), a *where* (per-asset GPS that places the camera, not just the EXIF), and a *when* (a stack of Mac-Absolute timestamps that distinguish capture from import from deletion) — and on iOS that catalog is captured live on the device, so it is far richer than the macOS library you already know.

## Why this matters

The camera roll is the single most evidentially dense store on an iPhone. A photo is rarely "just a picture": it carries a GPS fix that puts the *camera* at a coordinate to within a few metres, a capture timestamp with the original timezone, the bundle ID of the app that saved it (camera vs. Messages vs. a downloader), an on-device face cluster that links the subject to a *named person*, and an edit history that proves the original differs from what the user sees. `Photos.sqlite` knits all of that together and survives long after the user thinks a photo is gone — the 30-day Recently-Deleted window is recorded *in the database*, and derivative thumbnails outlive the original. You learned the macOS Photos library and `osxphotos` in the sibling course; the iOS schema is the close cousin, but because capture happens on-device, the location and people data are present and trustworthy in a way a desktop import often is not. This lesson is the practitioner map: the on-disk layout, the schema, how to resolve a row to a file on disk, and how to cross-corroborate a photo's claimed *where* against [[location-history]].

## Concepts

### The library on disk: `DCIM` (originals) + `PhotoData` (everything else)

The Photos library is not one folder — it is two sibling trees under `/private/var/mobile/Media/`:

```
/private/var/mobile/Media/
├── DCIM/                         ← ORIGINALS, the camera-roll capture tree
│   ├── 100APPLE/
│   │   ├── IMG_0001.HEIC
│   │   ├── IMG_0002.MOV
│   │   └── ...                   (rolls over to 101APPLE at 9999 files)
│   └── 101APPLE/
└── PhotoData/                    ← the catalog + every derived artifact
    ├── Photos.sqlite             ← THE catalog (Core Data)
    ├── Photos.sqlite-wal         ← write-ahead log (often holds the newest rows)
    ├── Photos.sqlite-shm
    ├── Thumbnails/V2/            ← assetsd-generated thumbnails (5005.JPG etc.)
    ├── Mutations/                ← edit recipes + rendered edited versions
    ├── CPLAssets/                ← iCloud Photo Library staging
    ├── AlbumsMetadata/
    └── protection/
```

The split is the first thing to internalise. `DCIM/` holds the **original master** of anything the device's own camera captured — the same DCIM convention every digital camera uses. Everything *else* the Photos subsystem produces — thumbnails, edited renders, video poster frames, the database itself — lives under `PhotoData/`. The `assetsd` daemon owns ingestion and the library; `photoanalysisd` / `mediaanalysisd` run the scene/face analysis that populates the people tables (and, critically, *only* when the device is idle, charging, and screen-off — see below).

> 🖥️ **macOS contrast:** On macOS the equivalent is `~/Pictures/<name>.photoslibrary/` — a *package* with `database/Photos.sqlite`, `originals/` (sharded `originals/D/D3FE…​.heic` by the first hex char of the UUID), and `resources/derivatives`. iOS uses the flat `DCIM/100APPLE` camera convention for originals instead of UUID-sharding, but the database, the daemon (`photoanalysisd` on both), and most table names are the same family. The tool you used in the macOS course, **`osxphotos`**, targets the *macOS* package layout — it will not run against a raw iOS `Photos.sqlite`; for iOS you reach for `iLEAPP` and the community SQL query sets.

### `Photos.sqlite`: a Core Data store, so think in `Z`-prefixes

`Photos.sqlite` is a SQLite database generated by Apple's **Core Data** ORM. That is why every table and column is `Z`-prefixed (`ZASSET`, `ZFILENAME`), every row has an integer primary key `Z_PK`, foreign keys are integer columns holding the other row's `Z_PK`, and a hidden `Z_PRIMARYKEY` / `Z_METADATA` pair records the schema. Many-to-many relationships (an asset in many albums) are stored in **join tables** named `Z_<NN>ASSETS` where `<NN>` is a *version-specific entity number* — it changes between iOS releases, so you discover it at query time rather than hard-coding it.

The single most important version tell: **the primary asset table was renamed `ZGENERICASSET` → `ZASSET` at iOS 14** (iOS 13 and earlier still show `ZGENERICASSET`), but the join tables, foreign-key column names, and even which columns exist drift *every* release. On the iOS-26-era target you will see `ZASSET`; on an older sample image you may see `ZGENERICASSET`. **Always run `PRAGMA table_info(...)` against the actual database before trusting any column name in this lesson** — Apple renames `ZDETECTEDFACE.ZASSET` to `ZDETECTEDFACE.ZASSETFORFACE` and back across versions, and a query written for one build silently returns nothing on another.

The tables you will join most, and how they hang off the asset spine:

```
                         ┌────────────────────────────┐
        1:1  ZASSET.Z_PK │ ZADDITIONALASSETATTRIBUTES  │  EXIF, orig filename,
        ◄────────────────┤   .ZASSET ─► ZASSET.Z_PK    │  tz, import attribution
                         └────────────────────────────┘
   ┌──────────┐  1:N  ┌───────────────────┐  N:1  ┌───────────┐
   │  ZASSET  │◄──────┤   ZDETECTEDFACE    ├──────►│  ZPERSON  │ named cluster
   │  (Z_PK)  │       │  .ZASSETFORFACE    │       │ + Contacts│
   │          │       │  .ZPERSONFORFACE   │       └───────────┘
   │          │  1:N  ├───────────────────┤
   │          │◄──────┤  ZINTERNALRESOURCE │ thumbnails, edits, posters
   │          │       │   .ZASSET          │ (everything but the master)
   │          │  N:N  ├───────────────────┤  N:N  ┌───────────────┐
   │          │◄──────┤   Z_<NN>ASSETS     ├──────►│ ZGENERICALBUM │
   └──────────┘       └───────────────────┘       └───────────────┘
```

> 🔬 **Forensics note:** **Copy before you query.** Opening `Photos.sqlite` with `sqlite3` — even a read-only `SELECT` — takes a write lock and can roll the WAL into the main file, changing hashes. Worse, the newest rows (the most recent photos) frequently still live in `Photos.sqlite-wal` and have not been checkpointed into the main file. Always copy **all three** (`Photos.sqlite`, `-wal`, `-shm`) together so the WAL replays consistently, work on the copy, and record the hash of the original first. A copy that omits the `-wal` can be missing the last hours of activity.

> 🔬 **Forensics note:** **Timestamps are Mac Absolute Time** — seconds since `2001-01-01 00:00:00 UTC`. Convert with `datetime(<col> + 978307200, 'unixepoch', 'localtime')`. This is the same epoch as `knowledgeC.db`/Biome (see [[knowledgec-db-deep-dive]], [[biome-and-segb-streams]]) and the [[the-ios-timestamp-zoo]] lesson catalogs every variant. The one exception in this store is `ZEXIFTIMESTAMPSTRING`, which is a *plain text string* baked from the camera's clock at capture — useful precisely because it preserves the device-local wall-clock independent of the row's UTC value.

### `ZASSET`: one row per photo or video

`ZASSET` is the spine. One row per asset; the columns you will use constantly:

| Column | Meaning / forensic use |
|---|---|
| `Z_PK` | Integer primary key — the join target for every other table |
| `ZUUID` | The asset's stable UUID (matches CloudKit / iCloud records) |
| `ZFILENAME` | On-disk filename, e.g. `IMG_0001.HEIC` |
| `ZDIRECTORY` | On-disk directory relative to `Media/`, e.g. `DCIM/100APPLE` |
| `ZDATECREATED` | **Capture time** (Mac Absolute) — when the photo was *taken* |
| `ZADDEDDATE` | When the asset was **added to this library** (import/AirDrop/save) |
| `ZMODIFICATIONDATE` | Last metadata change to the row |
| `ZLATITUDE` / `ZLONGITUDE` | GPS where the shot was taken; **`-180.0` is the "no location" sentinel** |
| `ZKIND` | `0` = photo, `1` = video |
| `ZKINDSUBTYPE` | Distinguishes screenshots, panoramas, Live Photos, etc. |
| `ZSAVEDASSETTYPE` | Provenance discriminator (`3` = captured by the device camera) |
| `ZHEIGHT` / `ZWIDTH` | Pixel dimensions |
| `ZDURATION` | Video length (seconds) |
| `ZFAVORITE` | User marked it a favourite |
| `ZHIDDEN` / `ZVISIBILITYSTATE` | In the Hidden album / hidden from the main grid |
| `ZTRASHEDSTATE` | `1` = in Recently Deleted (see below) |
| `ZTRASHEDDATE` | When it was trashed (Mac Absolute) — start of the 30-day clock |
| `ZAVALANCHEUUID` | Burst-group identifier; ties members of one burst together |
| `ZCLOUDPLACEHOLDERKIND` | Non-zero = the full-res original is in iCloud, only a thumbnail is local |

The `ZDATECREATED` vs. `ZADDEDDATE` split is one of the most useful facts in the whole store. For a photo the user took, the two are within seconds of each other. When `ZADDEDDATE` is *much later* than `ZDATECREATED`, the asset was **imported** — AirDropped, saved from Messages, downloaded, or restored from another device — and that gap is a provenance flag worth chasing.

> 🔬 **Forensics note:** `ZSAVEDASSETTYPE = 3` means the device's own camera captured it. Other values mark assets saved from elsewhere; pair `ZSAVEDASSETTYPE` with `ZADDITIONALASSETATTRIBUTES.ZIMPORTEDBYBUNDLEIDENTIFIER` (below) to attribute *which app* put the image on the device. The exact value→source mapping shifts between iOS versions — lead with the mechanism and confirm the enumeration against the specific build (Scott Koenig's and the Forensic Scooter query sets track the current values).

### Media subtypes: screenshots, Live Photos, bursts, panoramas

`ZKIND` only says photo (`0`) vs. video (`1`). The finer classification lives in **`ZKINDSUBTYPE`** (and a parallel bitmask the EXIF/`ZADDITIONALASSETATTRIBUTES` carries), which separates a normal photo from a **screenshot**, a **panorama**, a **Live Photo**, a **screen recording**, and so on. The exact integer values shift between releases — confirm them against the build — but the *distinction* is durable and forensically loaded:

- A **screenshot** (`ZKINDSUBTYPE` flags it; `ZSAVEDASSETTYPE` ≠ `3`) proves the user *captured what was on their own screen* — a message thread, a web page, a banking balance — not something they photographed in the world. It has no camera GPS and its `ZADDITIONALASSETATTRIBUTES.ZORIGINALFILENAME` looks like `IMG_####` but the EXIF lacks a `Make`/`Model`.
- A **Live Photo** is *two* files: the still (`IMG_####.HEIC`) **plus** a paired `.MOV` of the ~3 seconds around the shutter, tracked as a separate `ZINTERNALRESOURCE`. That motion clip can contain audio and frames the still does not — recover both.
- A **burst** ties its members together by a shared **`ZAVALANCHEUUID`**; the user's chosen "pick" is flagged, but every frame is retained on disk.

> 🔬 **Forensics note:** Screenshots are among the highest-value assets in a camera roll because they capture *content the user deliberately preserved from another app* — often the only surviving copy of a since-deleted message, a disappearing-media chat, or a transaction. Always segment the camera roll by subtype: `ZSAVEDASSETTYPE` + `ZKINDSUBTYPE` tells you camera-captured vs. screenshot vs. imported before you look at a single pixel.

### Resolving a row to its file on disk

The path is just concatenation: **`/private/var/mobile/Media/` + `ZDIRECTORY` + `/` + `ZFILENAME`**. A row with `ZDIRECTORY = 'DCIM/100APPLE'` and `ZFILENAME = 'IMG_0042.HEIC'` resolves to `/private/var/mobile/Media/DCIM/100APPLE/IMG_0042.HEIC`. That is the original master.

Two complications:

1. **The original may not be on the device.** With iCloud Photos + "Optimize iPhone Storage", `ZCLOUDPLACEHOLDERKIND` is non-zero and the only local file is a downscaled JPEG thumbnail in `PhotoData/Thumbnails/V2/` (the classic `5005.JPG`, generated by `assetsd` at ingest). The full-resolution original lives in iCloud and is a *cloud* acquisition problem (see [[icloud-acquisition-and-advanced-data-protection]]) — and if Advanced Data Protection is on, it is end-to-end encrypted and out of reach.
2. **Derivatives are tracked in `ZINTERNALRESOURCE`.** Every non-original file an asset owns — thumbnails, edited renders, video poster frames, adjusted versions — has a row in `ZINTERNALRESOURCE` joined back by `ZINTERNALRESOURCE.ZASSET = ZASSET.Z_PK`, with `ZRESOURCETYPE` describing the variant and `ZLOCALAVAILABILITY` telling you whether that file is actually present locally vs. iCloud-only. `ZINTERNALRESOURCE` tracks **everything except the original master** — so to enumerate *all* bytes belonging to an asset you read `ZASSET` (the original) **and** `ZINTERNALRESOURCE` (the rest).

### Thumbnails, the path map, and the analysis-state tell

Two on-disk companions are worth knowing. `assetsd` maintains **`PhotoData/Thumbnails/V2/`** — JPEG thumbnails (the `5005.JPG` family) generated at ingest — and **`PhotoData/assetUUIDForPath.plist`**, a plist mapping on-disk paths to asset UUIDs that helps reconcile orphaned files to catalog rows when the database is damaged or a file was carved. When the full-res original is iCloud-optimised away, that thumbnail is often the *only* viewable copy on the device, and it carries enough to identify the image even though it is not the master.

Per-asset enrichment progress is itself recorded: the **`ZASSETANALYSISSTATE`** table tracks how far `photoanalysisd`/`mediaanalysisd` got on each asset (scene classification, face detection, etc.). It is the machine-readable counterpart to the lazy-analysis caveat — an asset with no analysis state has not been processed, which explains an absent face row without implying the photo is faceless.

> 🔬 **Forensics note:** When `Photos.sqlite` is corrupt or partial, do not give up on orphaned media — `assetUUIDForPath.plist`, the `Thumbnails/V2` blobs, and the originals in `DCIM/` can be reconciled and parsed independently of the catalog. The bytes on disk are evidence even when the index that points at them is broken.

### `ZADDITIONALASSETATTRIBUTES`: EXIF detail, original filename, import attribution

A 1:1 sidecar table joined by `ZADDITIONALASSETATTRIBUTES.ZASSET = ZASSET.Z_PK`. It holds the richer, EXIF-derived and provenance detail that does not fit on the hot `ZASSET` row:

| Column | Meaning / forensic use |
|---|---|
| `ZORIGINALFILENAME` | The filename the asset had **before** Photos renamed it to `IMG_####`. For a download or a Messages save this is often the sender's filename or a UUID — a strong attribution lead |
| `ZEXIFTIMESTAMPSTRING` | Capture time as a **string in the device's local time at capture** — survives timezone arithmetic |
| `ZTIMEZONENAME` / `ZTIMEZONEOFFSET` | The capture timezone — lets you reconstruct local wall-clock and detect travel |
| `ZIMPORTEDBY` | Coarse import-source code |
| `ZIMPORTEDBYBUNDLEIDENTIFIER` | **Bundle ID of the app that saved the asset** (`com.apple.camera`, `com.apple.MobileSMS`, `com.apple.mobileslideshow`, a browser, a third-party app) — the cleanest single attribution field in the store |
| `ZCREATORBUNDLEID` | Bundle ID of the creating app where recorded |

> 🔬 **Forensics note:** `ZIMPORTEDBYBUNDLEIDENTIFIER` answers the question every camera-roll exam eventually faces: *did the suspect take this photo, or did it arrive?* `com.apple.camera` = shot on this device. `com.apple.MobileSMS` = saved out of Messages. A browser or third-party bundle = downloaded/sideloaded. Cross-reference with `ZSAVEDASSETTYPE`, the `ZDATECREATED`/`ZADDEDDATE` gap, and the EXIF in the file itself — they should agree, and disagreement is itself evidence (a stripped or mismatched EXIF on a "camera" asset is a tamper signal). MSAB's "media attribution for iOS" work walks this exact triangulation.

### Captions: user-authored text on a photo

A photo can carry a **caption** the user typed (the "Add a Caption" field in the info panel). That text is stored in the **`ZASSETDESCRIPTION`** table, reached from the asset through its attribute row (confirm the exact foreign key with `PRAGMA table_info(ZASSETDESCRIPTION)` — it has historically hung off `ZADDITIONALASSETATTRIBUTES` rather than `ZASSET` directly). A non-empty caption is **first-person, user-authored content** attached to a specific image at a specific time — among the strongest intent evidence the camera roll holds, because the user wrote it themselves.

> 🔬 **Forensics note:** Don't overlook captions in bulk parsing — most automated reports surface filenames and timestamps but bury the description field. A captioned photo can name a person, a place, or a circumstance the image alone would not establish. Query `ZASSETDESCRIPTION` explicitly.

### People: `ZDETECTEDFACE` and `ZPERSON` — on-device face clustering

This is where the iOS camera roll outclasses almost every other artifact for linking *people*. On-device ML (run by `photoanalysisd`/`mediaanalysisd`) detects faces, extracts a face *print* (an embedding), clusters prints that match the same individual, and surfaces the cluster as a **Person** the user can name.

```
ZASSET ──1:N──> ZDETECTEDFACE ──N:1──> ZPERSON
  (a photo)      (one box per face)      (a named/unnamed individual)
                       │
                       └── ZDETECTEDFACEPRINT (the face embedding)
                       └── ZFACECROP        (cropped face thumbnail on disk)
```

- **`ZDETECTEDFACE`** — one row per face *box* found in an asset. Joins to the asset by `ZDETECTEDFACE.ZASSETFORFACE = ZASSET.Z_PK` and to a person by `ZDETECTEDFACE.ZPERSONFORFACE = ZPERSON.Z_PK` **on recent schemas** — on older builds those columns are simply `ZASSET` / `ZPERSON` (this is exactly the kind of name that drifts; `PRAGMA table_info(ZDETECTEDFACE)` first). It also carries estimated attributes: `ZAGETYPE`, `ZGENDERTYPE`, `ZHAIRCOLORTYPE`, `ZBALDTYPE`, `ZGLASSESTYPE`, `ZFACIALHAIRTYPE`, plus the face's bounding box (`ZCENTERX`, `ZCENTERY`, `ZSIZE`) and a `ZQUALITYMEASURE`.
- **`ZPERSON`** — one row per clustered individual. `ZDISPLAYNAME` / `ZFULLNAME` hold the name **if the user named the cluster**, `ZFACECOUNT` how many faces belong to it, `ZTYPE`/`ZVERIFIEDTYPE` the confirmation state, `ZPERSONUUID` the stable id, and `ZCONTACTMATCHINGDICTIONARY` a serialized blob linking the cluster to a **Contacts entry** (name, phone, email) when Photos has matched the face to an address-book record.

> 🔬 **Forensics note:** A named `ZPERSON` plus its `ZDETECTEDFACE` rows lets you ask *"show me every photo containing this individual, and when/where each was taken"* — co-location and association evidence straight from the device, no manual review. `ZCONTACTMATCHINGDICTIONARY` can tie a face cluster to a phone number, bridging the camera roll to [[call-history-voicemail-contacts-interactions]] and [[communications-imessage-and-sms]]. **Caveat: the estimated attributes are model output, not ground truth** — `ZAGETYPE`/`ZGENDERTYPE` are coarse classifier guesses and have been wrong in casework; treat them as leads, never as fact, and never assert an age from `ZAGETYPE` in a report.

> ⚠️ **ADVANCED:** The people tables are **populated lazily** — `photoanalysisd` only runs analysis when the device is **idle, charging, and screen-off**. A freshly captured photo, or a phone seized while in heavy use, may have *no* face rows yet even for obvious faces. Empty `ZDETECTEDFACE`/`ZPERSON` is therefore **not** evidence the photos contain no people; it can simply mean analysis never got an idle-charging window. (You will see this concretely in the Simulator lab — `Photos.sqlite` there has assets but zero detected faces, because the Simulator does not run `photoanalysisd` at all.)

### Albums: `ZGENERICALBUM` + a version-numbered join table

`ZGENERICALBUM` is one row per album — user albums, smart albums, system collections (Recents, Recently Deleted, Hidden). Key columns: `ZTITLE` (album name), `ZKIND` (user vs. smart vs. system), `ZUUID`. Membership is the many-to-many join table `Z_<NN>ASSETS`, whose `<NN>` is version-specific (you may see `Z_23ASSETS`, `Z_30ASSETS`, etc.). The join table's own columns are also numbered — e.g. a `Z_23ASSETS` table with columns `Z_23ALBUMS` and `Z_30ASSETS` — so you discover both at query time:

```sql
-- find the join table and its columns dynamically
SELECT name FROM sqlite_master WHERE type='table' AND name LIKE 'Z_%ASSETS';
-- then PRAGMA table_info(Z_23ASSETS); to see the two FK columns
```

> 🔬 **Forensics note:** A *user-created* album named for a person, place, or topic is intent evidence — the user deliberately curated those assets. Smart/system albums (`ZKIND` marks them) are auto-generated and carry no such intent. Distinguish the two before drawing conclusions about curation.

### The Syndication library: photos that arrived but were never saved

There is a **second, separate Photos library** that most camera-roll exams miss. When media is shared into the device through **"Shared with You"** — primarily images sent in Messages that the user *never tapped Save on* — it is catalogued in the **Syndication Photo Library**, not the local camera roll. It has its own `Photos.sqlite` (the same schema family) under:

```
/private/var/mobile/Library/Photos/Libraries/Syndication.photoslibrary/database/Photos.sqlite
   originals (FFS):  …/Syndication.photoslibrary/scopes/syndication/*
   logical backup:   /private/var/mobile/Media/PhotoData/UBF/scopes/syndication/*
```

The forensic payoff is large: this is **media the user received and possibly viewed but deliberately did not save** — exactly the content a suspect assumes is gone because it is not in the camera roll. The Syndication `Photos.sqlite` links each asset back to the originating Messages conversation, and `ZADDITIONALASSETATTRIBUTES.ZORIGINALFILENAME` plus `ZIMPORTEDBYBUNDLEIDENTIFIER = com.apple.MobileSMS` ties it to the sender. When a user *does* save a Shared-with-You asset to the camera roll (or deletes it from view), the **local** `Photos.sqlite` records `ZSAVEDASSETTYPE = 12`, marking that the asset was, at some point, a Shared-with-You syndicated link.

> 🔬 **Forensics note:** Always pull **both** `Photos.sqlite` databases — the local PhotoData one *and* the Syndication one — and the `UBF/scopes/syndication` tree. A photo absent from the camera roll may still be sitting in the syndication library, with its Messages provenance intact. Scott Koenig's `PhotoData-Synd-Photos.sqlite_Queries` repo is built precisely to query both with one query set; this artifact is current as of iOS 16+/17+ and only grows in relevance with disappearing-message features.

### Recently Deleted and the 30-day window

When a user deletes a photo it is **not** removed — `Photos.sqlite` sets `ZTRASHEDSTATE = 1` and stamps `ZTRASHEDDATE`, the file stays on disk, and the asset moves into the "Recently Deleted" album for **30 days**, after which `assetsd` purges row and file. That window is a gift: a photo deleted up to 30 days before seizure is *fully recoverable* — the original bytes are still in `DCIM/`, and the row tells you exactly when the user tried to delete it.

```sql
SELECT ZFILENAME,
       datetime(ZTRASHEDDATE + 978307200,'unixepoch','localtime') AS trashed,
       date(ZTRASHEDDATE + 978307200 + 30*86400,'unixepoch','localtime') AS purges_on
FROM   ZASSET                 -- ZGENERICASSET on iOS-13-and-earlier schemas
WHERE  ZTRASHEDSTATE = 1
ORDER  BY ZTRASHEDDATE DESC;
```

> 🔬 **Forensics note:** `ZTRASHEDDATE` is an **intent timestamp** — the moment the user chose to delete this specific image. Correlated against a known event (a message thread, a search, a power-off) it can be highly probative. After the 30-day purge the row is gone from `Photos.sqlite`, but fragments may survive in the `-wal`, in older iCloud records, or be carvable — see [[deleted-data-recovery]].

### Edits: the original is preserved, the edit is a recipe

Photos edits are **non-destructive**. The original master in `DCIM/` is never overwritten. When a user edits, `ZASSET.ZHASADJUSTMENTS` is set to `1`, the edit *recipe* (crop, filters, markup, the adjustment graph) is written as a plist under `PhotoData/Mutations/.../Adjustments/`, and a fully-rendered edited version is written alongside it (and tracked in `ZINTERNALRESOURCE`). So for any edited asset you can recover **both** what the camera captured **and** what the user chose to show.

> 🔬 **Forensics note:** The original-vs-edited divergence is evidence in itself. A user who crops a person, a location sign, or a timestamp out of a photo leaves the *uncropped original* in `DCIM/` and the crop recipe in the mutation plist — you can reconstruct exactly what was removed and when. Confirm the exact mutation subpath on your target build (`PhotoData/Mutations/` has carried versioned subfolders that shift between releases) before scripting extraction.

### Synthesis: who / where / when — and corroborating the *where*

Put the pieces together and one asset answers all three:

```
 WHO   ZDETECTEDFACE → ZPERSON (named) → ZCONTACTMATCHINGDICTIONARY → a contact
 WHERE ZASSET.ZLATITUDE / ZLONGITUDE  (camera position, not a network guess)
 WHEN  ZDATECREATED (taken) · ZADDEDDATE (added) · ZTRASHEDDATE (deleted)
       + ZEXIFTIMESTAMPSTRING / ZTIMEZONENAME (local wall-clock + tz)
 HOW   ZSAVEDASSETTYPE + ZIMPORTEDBYBUNDLEIDENTIFIER (camera vs. imported, by which app)
```

The highest-value move is **cross-referencing the photo's GPS against the device's own location history.** A photo's `ZLATITUDE/ZLONGITUDE` claims the *camera* was at a coordinate at `ZDATECREATED`. The `routined` daemon's significant-locations store independently logs where the *device* was at that time. When the two agree, the camera was the device and the user was present; when a photo's GPS sits far from where `routined` places the phone at that minute, the asset was very likely **imported, not captured here** — a discrepancy that exposes a planted or transferred image. That is the kind of corroboration that turns a single artifact into a defensible finding (see [[location-history]] and the timeline build in [[building-a-unified-timeline]]).

> ⚖️ **Authorization:** Camera-roll GPS is precise enough to place a person at a private address, a clinic, a protest, or a victim's location. Treat per-asset coordinates with the same care as any tracking data — pull only assets within the scope of your authority, document the query that selected them, and remember that face clusters identify *third parties* who are not your subject. Minimisation applies to the people in the photos, not just the device owner.

## Hands-on

There is no shell on the device — every command runs on your Mac against an extracted library, a backup, a sample image, or the **Simulator** (whose `data/Media/` tree is a structural mirror of the device's `/private/var/mobile/Media/`). Always copy the DB + sidecars first.

**Locate and safely copy the catalog (Simulator):**

```bash
# Find every Photos.sqlite across installed simulators
find ~/Library/Developer/CoreSimulator/Devices \
     -path '*/data/Media/PhotoData/Photos.sqlite' 2>/dev/null

DB=~/Library/Developer/CoreSimulator/Devices/<UDID>/data/Media/PhotoData/Photos.sqlite
WORK=/tmp/photos_case
mkdir -p "$WORK"
# copy ALL three so the WAL replays consistently — never just the .sqlite

shasum -a 256 "$DB" | tee "$WORK/original.sha256"
```

**The core asset query — who/where/when/how in one shot:**

```bash
sqlite3 "$WORK/Photos.sqlite" "
SELECT a.Z_PK,
       a.ZFILENAME,
       a.ZDIRECTORY,
       datetime(a.ZDATECREATED + 978307200,'unixepoch','localtime') AS taken,
       datetime(a.ZADDEDDATE   + 978307200,'unixepoch','localtime') AS added,
       a.ZLATITUDE, a.ZLONGITUDE,
       a.ZKIND, a.ZSAVEDASSETTYPE, a.ZTRASHEDSTATE,
       aa.ZORIGINALFILENAME,
       aa.ZEXIFTIMESTAMPSTRING,
       aa.ZTIMEZONENAME,
       aa.ZIMPORTEDBYBUNDLEIDENTIFIER
FROM   ZASSET a
LEFT   JOIN ZADDITIONALASSETATTRIBUTES aa ON aa.ZASSET = a.Z_PK
ORDER  BY a.ZDATECREATED DESC
LIMIT  25;" -header -column
```

(If `.tables` shows `ZGENERICASSET` instead of `ZASSET`, swap the table name — that DB is an iOS-13-or-earlier schema.)

**A one-asset dossier** — collapse who/where/when/how for a single `Z_PK` into one row you can paste into a report:

```bash
PK=42   # the asset's Z_PK
sqlite3 "$WORK/Photos.sqlite" "
SELECT a.ZUUID,
       a.ZDIRECTORY || '/' || a.ZFILENAME                       AS path,
       datetime(a.ZDATECREATED + 978307200,'unixepoch','localtime') AS taken,
       datetime(a.ZADDEDDATE   + 978307200,'unixepoch','localtime') AS added,
       CASE WHEN a.ZTRASHEDSTATE=1
            THEN datetime(a.ZTRASHEDDATE+978307200,'unixepoch','localtime')
            ELSE 'live' END                                      AS trashed,
       a.ZLATITUDE, a.ZLONGITUDE,
       aa.ZEXIFTIMESTAMPSTRING, aa.ZTIMEZONENAME,
       aa.ZORIGINALFILENAME, aa.ZIMPORTEDBYBUNDLEIDENTIFIER,
       a.ZHASADJUSTMENTS                                         AS edited
FROM   ZASSET a
LEFT   JOIN ZADDITIONALASSETATTRIBUTES aa ON aa.ZASSET = a.Z_PK
WHERE  a.Z_PK = $PK;" -line
```

On iOS the same *questions* are answered by raw `sqlite3` + `exiftool` + `iLEAPP`; there is no `osxphotos` because the on-disk package layout differs. The SQL muscle memory transfers directly — same epoch math, same `Z_PK` joins, mostly the same column names.

## 🧪 Labs

> ⚠️ Every lab copies the database (plus `-wal`/`-shm`) before querying. The `cp` is mandatory and non-negotiable — querying the live store mutates it.

### Lab 1 — Dissect the schema on the Simulator (substrate: **Xcode Simulator / CoreSimulator**)

**Fidelity caveat:** the Simulator runs macOS frameworks — **no SEP, no Data-Protection-at-rest, no `photoanalysisd`/`mediaanalysisd`.** You get the *real* `Photos.sqlite` schema and the `DCIM`/`PhotoData` layout (ground truth for structure and path resolution), but device-only enrichment never runs: **`ZDETECTEDFACE` and `ZPERSON` will be empty** even with faces in the photos. Use this lab for layout, joins, and timestamp math; use Lab 2 for the people/location enrichment the Simulator cannot produce.

1. `find` the Simulator's `Photos.sqlite`, copy it and its `-wal`/`-shm` to `/tmp/photos_case`, and record the SHA-256.
2. `xcrun simctl addmedia booted` a couple of images, then re-run the core asset query. Confirm new `ZASSET` rows, and note `ZDATECREATED` vs. `ZADDEDDATE` for an *added* (not captured) image — observe the gap that flags an import.
3. Resolve every row to a path with the path-reconstruction loop and verify the files exist under `data/Media/DCIM/`.
4. Run `SELECT COUNT(*) FROM ZDETECTEDFACE;` — confirm it returns **0**, and write one sentence explaining *why* an empty faces table is not evidence of a faceless library.

### Lab 2 — People, GPS, and Recently-Deleted on a real device image (substrate: **public sample forensic image**)

**Fidelity caveat:** uses Josh Hickman's iOS reference image (thebinaryhick.blog / Digital Corpora) or the iLEAPP test data — a *real* device library where `photoanalysisd` did run, so the enrichment the Simulator lacks is present.

1. Acquire and verify the sample image's hash; extract `PhotoData/Photos.sqlite` (+ `-wal`/`-shm`) read-only.
2. List named persons (`ZPERSON.ZDISPLAYNAME IS NOT NULL`) and, for one named person, pull every asset they appear in with capture time and GPS. You have just produced a co-location timeline for an individual from the camera roll alone.
3. Run the Recently-Deleted query (`ZTRASHEDSTATE = 1`). For each, compute the purge date and confirm the original bytes still resolve on disk — these are recoverable deletions.
4. Pick one geotagged asset and note its coordinate and `ZDATECREATED`. (Lab 3 corroborates it.)

### Lab 3 — Corroborate a photo's GPS against location history (substrate: **read-only walkthrough + sample image**)

**Fidelity caveat:** `routined`'s significant-locations store does not exist in the Simulator; this uses the same sample image as Lab 2.

1. Take the geotagged asset from Lab 2 (coordinate + `ZDATECREATED`).
2. Open the device's location store and query for fixes within ±15 minutes of `ZDATECREATED` (the daemon, paths, and schema are covered in [[location-history]]).
3. Compute the distance between the photo's GPS and the nearest `routined` fix. **Agreement** corroborates that the device captured the photo and the owner was present; a large gap argues the asset was **imported, not shot here** — cross-check `ZIMPORTEDBYBUNDLEIDENTIFIER` and the `ZDATECREATED`/`ZADDEDDATE` gap to confirm.
4. Write the two-line finding you would put in a report, citing the asset `ZUUID`, both timestamps, and the distance.

### Lab 4 — Find what was received but never saved (substrate: **public sample forensic image**)

**Fidelity caveat:** the Syndication library is populated by real Messages "Shared with You" activity, which the Simulator does not exercise — use a sample FFS image that includes Messages traffic.

1. Locate the Syndication library at `/private/var/mobile/Library/Photos/Libraries/Syndication.photoslibrary/database/Photos.sqlite` in the extraction; copy it + sidecars.
2. Run the Syndication query above and list assets present here but **not** in the local `Photos.sqlite` (compare `ZUUID` sets). These are received-but-unsaved media.
3. For one such asset, read `ZORIGINALFILENAME` and `ZIMPORTEDBYBUNDLEIDENTIFIER`, then pivot to [[communications-imessage-and-sms]] to find the conversation it arrived in.
4. In the *local* library, count assets with `ZSAVEDASSETTYPE = 12` — these were once Shared-with-You links the user *did* save. Contrast the two populations.

## Pitfalls & gotchas

- **Querying without the `-wal` loses the newest photos.** Recent rows are frequently still in `Photos.sqlite-wal` and not yet checkpointed. Copy all three files together; a lone `.sqlite` can silently omit the last hours.
- **Hard-coding column or join-table names.** `ZGENERICASSET` vs. `ZASSET`, `ZDETECTEDFACE.ZASSET` vs. `ZASSETFORFACE`, `Z_23ASSETS` vs. `Z_30ASSETS` — all drift by version. `PRAGMA table_info(...)` / `SELECT name FROM sqlite_master` first, every time, or your query returns empty and you wrongly conclude "no data".
- **Wrong epoch.** These are Mac Absolute Time (2001). Forgetting the `+ 978307200` yields timestamps ~31 years off; mixing in Unix or WebKit epochs from other stores corrupts a timeline. See [[the-ios-timestamp-zoo]].
- **`ZLATITUDE = -180.0` is "no GPS", not the South Pole at the antimeridian.** Filter the `-180.0` sentinel out before mapping, or you will plot bogus points.
- **Empty `ZDETECTEDFACE`/`ZPERSON` ≠ no people.** Analysis only runs idle+charging+screen-off; a busy or freshly-seized phone may have unanalysed photos. Never report "no faces detected" as "no people in photos".
- **Treating estimated face attributes as fact.** `ZAGETYPE`/`ZGENDERTYPE` are coarse classifier output and have been wrong. Leads only — never assert an age or gender from them.
- **Logical backups exclude the camera-roll media.** A standard Finder/iTunes backup omits the `DCIM` originals (and much of `PhotoData`); the rich `Photos.sqlite` story needs a **full file-system acquisition** ([[full-file-system-acquisition]], [[the-itunes-finder-backup-format]]). Don't conclude "few photos" from a logical backup.
- **iCloud-optimised originals aren't local.** `ZCLOUDPLACEHOLDERKIND` non-zero means only a thumbnail is on the device; the original is a cloud-acquisition problem, blocked entirely under ADP.
- **EXIF in the file can disagree with the DB.** Apple may strip or alter embedded EXIF; the DB is authoritative for library state, the file's EXIF for what the camera baked in. Disagreement is a finding, not a bug — record both.
- **Forgetting the second library.** Media absent from the camera roll may live in the **Syndication** `Photos.sqlite` (Shared-with-You). Examining only `PhotoData/Photos.sqlite` misses everything the user received in Messages and never saved.
- **Screenshots have no camera GPS or `Make`/`Model` — that is expected.** Don't flag a geotag-less `IMG_####` as suspicious until you check `ZKINDSUBTYPE`; a screenshot legitimately lacks both.

## Key takeaways

- `Photos.sqlite` (`/private/var/mobile/Media/PhotoData/`) is a Core Data catalog; `ZASSET` (formerly `ZGENERICASSET`) is one row per photo/video, joined by integer `Z_PK` to attribute, face, resource, and album tables.
- Originals live in `DCIM/100APPLE/`; everything derived (thumbnails, edits, the DB) lives in `PhotoData/`. Resolve a row with `Media/ + ZDIRECTORY + / + ZFILENAME`; enumerate the rest via `ZINTERNALRESOURCE`.
- The *when* is three distinct Mac-Absolute timestamps — `ZDATECREATED` (taken), `ZADDEDDATE` (added), `ZTRASHEDDATE` (deleted) — plus `ZEXIFTIMESTAMPSTRING`/timezone; the gap between created and added flags imports.
- The *where* is per-asset `ZLATITUDE/ZLONGITUDE` placing the *camera* (with `-180.0` as the no-fix sentinel); corroborate it against `routined` location history to separate captured-here from imported.
- The *who* is on-device face clustering: `ZDETECTEDFACE → ZPERSON`, with named clusters and `ZCONTACTMATCHINGDICTIONARY` bridging to Contacts — but it only populates when idle+charging, and its attribute estimates are leads, not facts.
- Provenance/attribution lives in `ZSAVEDASSETTYPE` + `ZADDITIONALASSETATTRIBUTES.ZIMPORTEDBYBUNDLEIDENTIFIER` + `ZORIGINALFILENAME`: camera vs. Messages vs. download, named by bundle ID.
- Deleted photos persist for 30 days (`ZTRASHEDSTATE=1`, `ZTRASHEDDATE`), fully recoverable from disk; edits are non-destructive (original kept, recipe in `PhotoData/Mutations/`).
- There are **two** Photos libraries: the local camera roll and a separate **Syndication** library for Shared-with-You media received but never saved — pull both, plus segment the roll by subtype (screenshots carry content from other apps).
- Copy `.sqlite` + `-wal` + `-shm` before any query, `PRAGMA` before trusting a column name, and acquire full-file-system — logical backups omit the media.

## Terms introduced

| Term | Definition |
|---|---|
| `Photos.sqlite` | The Core Data catalog of the iOS Photos library at `/private/var/mobile/Media/PhotoData/`; the master index of every asset |
| `DCIM/100APPLE` | The camera-roll **originals** tree under `Media/`; rolls over to `101APPLE` at 9999 files |
| `PhotoData/` | Sibling tree holding the database, thumbnails, edits, and all derived artifacts |
| `ZASSET` | Primary asset table — one row per photo/video (named `ZGENERICASSET` through iOS 13; `ZASSET` from iOS 14) |
| `Z_PK` | Core Data integer primary key; the join target for every foreign-key column |
| `ZADDITIONALASSETATTRIBUTES` | 1:1 sidecar with EXIF detail, original filename, capture timezone, and import attribution |
| `ZINTERNALRESOURCE` | Tracks every file an asset owns *except* the original master (thumbnails, edits, posters) |
| `ZDETECTEDFACE` | One row per detected face box; joins an asset to a person; carries estimated face attributes |
| `ZPERSON` | One row per clustered individual; holds the user-assigned name and a Contacts-matching blob |
| `ZCONTACTMATCHINGDICTIONARY` | Serialized link from a face cluster to an address-book contact (name/phone/email) |
| `ZGENERICALBUM` | One row per album; membership via a version-numbered `Z_<NN>ASSETS` join table |
| `ZTRASHEDSTATE` / `ZTRASHEDDATE` | `1` + timestamp marking an asset in the 30-day Recently-Deleted window |
| `ZSAVEDASSETTYPE` | Provenance discriminator (`3` = captured by this device's camera) |
| `ZIMPORTEDBYBUNDLEIDENTIFIER` | Bundle ID of the app that saved the asset — the cleanest attribution field |
| `ZCLOUDPLACEHOLDERKIND` | Non-zero ⇒ the full-res original is in iCloud and only a local thumbnail exists |
| `ZKINDSUBTYPE` | Fine media classification (screenshot, panorama, Live Photo, screen recording) beyond `ZKIND` |
| `ZAVALANCHEUUID` | Burst-group identifier tying every frame of one burst together |
| Syndication library | A second `Photos.sqlite` (`Syndication.photoslibrary`) cataloguing Shared-with-You media received (e.g. in Messages) but not saved to the camera roll |
| `ZSAVEDASSETTYPE = 12` | Local-library marker that an asset was, at some point, a Shared-with-You syndicated link |
| Mac Absolute Time | Epoch of `2001-01-01 UTC`; convert with `+ 978307200` to Unix seconds |
| `photoanalysisd` / `assetsd` | Daemons that run face/scene analysis and own library ingestion, respectively |
