# FileVault & Encryption Internals

> **In one sentence:** FileVault on Apple Silicon is not a traditional full-disk encryption bolt-on — it is the point at which the SEP releases a hardware-entangled key hierarchy that was protecting your data all along, and understanding that distinction changes everything about how you reason about acquisition, key rotation, and what "erase" actually means.

---

## Why this matters

If you migrated from Windows BitLocker, you are carrying a mental model that needs surgery. BitLocker is a software layer that _adds_ encryption to an unencrypted storage subsystem. FileVault 2 on Apple Silicon is the _access control gate_ on a storage subsystem that is **already always encrypted** at the hardware layer. The implications cascade:

- "Enabling FileVault" does not start encrypting data — it changes _who can authorize the key release at boot_.
- "Disabling FileVault" does not decrypt the volume — it changes the boot policy so the key is released automatically without a user credential.
- "Erasing" a volume is instantaneous and cryptographically complete the moment one key is discarded — no overwrite passes needed, no secure-erase ioctl, no time.
- Cold acquisition of a powered-off Apple Silicon Mac without the password or recovery key yields **astronomically expensive ciphertext**, not a filesystem.

This is the deepest architectural difference between the Mac and every x86 Windows machine you have ever imaged.

---

## Concepts

### 1. The APFS Volume Model: System vs. Data

macOS uses a **dual-volume** APFS container structure introduced in macOS Catalina:

```
APFS Container (disk0s2)
├── Macintosh HD  [System Volume — SSV, signed, sealed, read-only at runtime]
│   └── Mounted read-only; cryptographically signed by Apple (Signed System Volume)
└── Macintosh HD - Data  [Data Volume — encrypted, writable]
    └── THIS is what FileVault actually governs
```

The System volume is sealed with a SHA-256 Merkle hash tree (`csrutil authenticated-root status` shows this). It is also encrypted on Apple Silicon but via a different, Apple-managed key — users have no control over it and it is irrelevant to FileVault configuration. The **Data volume** holds your home directory, applications, and all user-generated content. It is the target of all FileVault operations.

> 🪟 **Windows contrast:** BitLocker encrypts the entire volume including the OS. Apple split the OS into a signed, read-only, separately-keyed volume — the equivalent of having Windows ship as a tamper-evident ROM image. This also means Apple Silicon can verify OS integrity independently of user data encryption state.

---

### 2. The Key Hierarchy: From Silicon to Password

Understanding the hierarchy requires naming four actors:

| Actor | What it is |
|---|---|
| **UID** | A 256-bit AES key fused into the SoC die at manufacture time. Never leaves the Secure Enclave. Apple does not have a copy. |
| **SEP** (Secure Enclave Processor) | A co-processor inside every Apple Silicon SoC (and the T2 on Intel). Runs its own signed firmware (`sepOS`). Manages all cryptographic operations below the OS layer. |
| **VEK** (Volume Encryption Key) | The symmetric key (AES-256-XTS) that directly encrypts every block of the Data volume. |
| **KEK** (Key Encryption Key) | A key that wraps the VEK. Derived from user credential + hardware UID. Multiple KEKs can protect a single VEK. |

The flow, from bottom to top:

```
Hardware UID (fused, SEP-only)
    │
    ├─ + xART (Secure Enclave anti-replay token, per-volume random)
    │
    └─→ VEK (generated by SEP, protects all data blocks via AES Engine)
              │
              └─ wrapped by one or more KEKs:
                    ├── KEK_user_1  ← derived from (user password + UID), stored in Volume Keybag
                    ├── KEK_user_2  ← additional enabled user (fdesetup list)
                    ├── KEK_recovery ← personal recovery key (PRK), 28-char Base32
                    └── KEK_institutional ← MDM/enterprise, optional
```

The VEK and its wrapping KEKs are stored in two on-disk keybag structures (both encrypted):

- **Container Keybag** (`/private/var/db/SystemKey` on older toolchains; now accessed via `diskutil apfs listcryptoUsers /`) — contains wrapped VEKs for every encrypted volume in the APFS container.
- **Volume Keybag** (embedded in the APFS metadata on the volume itself) — contains the KEK records that correspond to each authorized unlock credential.

Key wrapping uses **RFC 3394 AES Key Wrap** — the KEK is never stored in plaintext anywhere accessible to macOS. The SEP performs all unwrap operations internally. The CPU never sees the raw VEK.

> 🔬 **Forensics note:** The keybag structures are on-disk, readable as raw bytes in a full disk image. But they are only useful if you can unwrap them — which requires either the user password, the PRK, or the institutional key. Without one of those, you have encrypted blobs backed by a hardware UID that died with the specific SoC. No rainbow table, no brute-force on commodity hardware applies to the UID-entangled wrapping.

### 3. Apple Silicon vs. Intel/T2: The Critical Distinction

**Apple Silicon (M1 and later, including all current hardware):**

- The AES Engine sits directly on the memory bus. Encryption/decryption is transparent to all reads and writes — there is no software encryption path.
- The SEP controls key material. The OS _requests_ decryption; the SEP either grants or denies based on boot policy and authentication state.
- Data is encrypted at rest **regardless of FileVault state**. Disabling FileVault changes only the boot-time authorization behavior (auto-unlock vs. password-gated unlock). The bits on the NAND are always ciphertext.
- The **xART** (eXtended Anti-Replay Token) is a hardware counter/nonce that prevents replay attacks — even if you extracted the raw NAND and cloned it, the xART state would not match and the SEP would refuse key release.

**Intel with T2 chip (2018–2020 MacBook Pro/Air/mini/iMac Pro):**

- Same conceptual model as Apple Silicon — T2 is functionally the SEP for storage. Hardware AES encryption always on.
- FileVault behavior: same gate mechanism, different silicon path.
- Key difference: T2 Macs can sometimes be targeted by DFU-mode exploits (checkm8 derivatives) that were demonstrated in the 2020–2022 timeframe. The T2's sepOS had exploitable vulnerabilities that no longer exist on Apple Silicon's more isolated SEP architecture. On Apple Silicon, there is no known DFU-mode key extraction path.

**Intel without T2 (pre-2018):**

- No hardware AES engine on the storage path. FileVault 2 here is _genuinely software encryption_ via the CoreStorage/APFS layer. Data is only encrypted when FileVault is on.
- The key hierarchy still exists but is protected only by a software-managed keybag, not a co-processor. A memory forensics capture of a running machine can potentially expose key material in RAM.
- These machines are effectively end-of-life for security-critical deployment; noting them only for legacy acquisition contexts.

> 🪟 **Windows contrast:** BitLocker with TPM 2.0 is architecturally closest to the T2 model. Apple Silicon's SEP is significantly more isolated than a TPM — it runs its own OS, has its own secure boot chain, and handles more of the authentication logic directly rather than delegating to the main CPU.

### 4. What FileVault Actually Controls

Given that the Data volume is always encrypted, enabling FileVault changes exactly one policy: **whether the boot process requires a user password to release the KEK before the OS loads**.

| State | At boot | Data at rest |
|---|---|---|
| FileVault **off** | Boot policy instructs SEP to auto-release the key. System mounts volume without prompting. | Encrypted (VEK active, no password gate) |
| FileVault **on** | Pre-boot environment (recoveryOS sub-layer) prompts for password. SEP unwraps KEK only if credential matches. | Encrypted + password-gated |

The pre-boot authentication screen runs before the full macOS kernel loads. It is part of the **LocalAuthentication** framework in its reduced recoveryOS form. This is why you cannot use Touch ID at the FileVault pre-boot screen — biometrics require the full OS security framework.

**Who can unlock** is tracked per-user. `fdesetup list` shows all enabled users. A machine may have FileVault on but only one of four local accounts may be in the unlock list — the others would need to log in after the first-unlocked boot.

### 5. The Recovery Key

Every FileVault enable operation should produce a **Personal Recovery Key (PRK)** — a 28-character Base32 string in the format `XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XX`. This is a KEK credential, not the VEK itself. Possessing it means you can unlock the volume even without any local user's password.

**Recovery key escrow options:**

| Option | How | Where stored | Who has access |
|---|---|---|---|
| iCloud escrow | Automatic when signed into iCloud during enable | Apple's servers, encrypted to your iCloud key | You (via iforgot.apple.com) + Apple under lawful process |
| Self-held PRK | Write it down / store in password manager | Wherever you put it | You only |
| Institutional key | MDM (e.g., Jamf) deploys an asymmetric key pair | MDM server | IT / MDM admin |

> 🔬 **Forensics note:** The iCloud-escrowed recovery key is the primary **lawful-access path** for Apple on FileVault-encrypted Macs. Apple's Transparency Report documents government requests for iCloud data, which can include escrowed FileVault keys. For corporate investigation, MDM-held institutional keys are the operational path. For a self-managed machine with a PRK known only to the user, there is no practical key-recovery path without the password or PRK — Apple cannot help.

**macOS Tahoe behavior change:** As of macOS 26 (Tahoe), `sudo fdesetup enable` no longer prints the PRK to stdout. The key is created but silently stored. Use the `-outputplist` flag to capture it programmatically:

```bash
sudo fdesetup enable -outputplist > ~/Desktop/filevault-key.plist
# The plist contains RecoveryKey, EnabledUser, HardwareUUID, SerialNumber
# Move this somewhere secure immediately; delete the Desktop copy
```

### 6. Cryptographic Erase: Instant, Complete, Irreversible

The **"Erase All Content and Settings"** operation (and its CLI equivalent `eraseassistant` / `--eraseinstall`) performs a **cryptographic erase (CE)**, not a wipe:

1. The SEP deletes (zeroes) the VEK and the xART from its internal secure storage.
2. The keybag structures on disk now contain KEK records that wrap a key that no longer exists.
3. All NAND blocks that held the volume's ciphertext are now permanently inaccessible — mathematically equivalent to overwriting with random data, because the decryption key is gone.
4. The volume is reformatted with new empty APFS structures and a new VEK generated by the SEP.

Total time: seconds. Total overwrite passes: zero.

This is why "secure erase" passes (`rm -P`, `shred`, `dd if=/dev/urandom`) are meaningless on Apple Silicon — and why macOS itself no longer offers them in Disk Utility. The Secure Erase Free Space option was removed because it provided false assurance without actual benefit on encrypted NAND.

> ⚠️ **ADVANCED / DESTRUCTIVE:** Cryptographic erase is **instantaneous and complete**. There is no undo, no recycle bin, no "oops" path. Before any CE operation: back up via Time Machine, confirm the backup restored to a test machine, and store the PRK or iCloud account credentials somewhere you will still have after the Mac is blank.

> 🔬 **Forensics note:** Research has explored APFS deleted-file recovery using tools like `apfs_util` on machines where the key is known (e.g., a live imaged machine). On a properly CE'd machine without the old VEK, no such recovery is possible — there are no plaintext blocks to recover, and the old keybag is useless without the destroyed VEK. The distinction between "CE with known key" and "CE with destroyed key" is critical in investigation planning.

### 7. Encrypted Disk Images and Sparsebundles

Beyond full-volume FileVault, macOS provides **encrypted disk image** containers that work on any filesystem and are independent of device-level encryption:

| Format | Use case | Notes |
|---|---|---|
| `.dmg` (read-only encrypted) | Software distribution, archive vaults | `hdiutil create -encryption AES-256 -stdinpass` |
| `.sparsebundle` | Time Machine, large mutable vaults | Grows on demand; uses band files for efficient backup |
| `.sparseimage` | Single-file sparse encrypted image | Older format; still valid |

These use **AES-256-CBC** (for AES-256 encryption images; AES-128-CBC for the legacy AES-128 option) with a passphrase-derived key, **not** SEP-backed. The keychain can cache the passphrase so you do not re-enter it on every mount. These are software-only encryption containers — the password is the only protection.

> 🔬 **Forensics note:** Encrypted DMGs and sparsebundles are common artifacts in Mac investigations. Their headers are identifiable (magic bytes `0x78 0x61 0x72 0x21` for UDIF images). The encryption is generally strong; password attack via hashcat/john using extracted hash is the primary vector. `hdiutil imageinfo` on a mounted image reveals encryption method and key length. An unmounted encrypted image is opaque — mount is required for any content analysis.

### 8. Cryptomator: Cloud-Side Encryption

For data stored in iCloud Drive, Dropbox, or any cloud sync service, **Cryptomator** (open source, MIT license) provides **client-side, SEP-independent encryption**:

- Creates a "vault" directory that looks like a folder to the sync service but contains individually-encrypted files.
- Uses AES-256-SIV for file content and AES-256-SIV for file name encryption, with a Scrypt-derived master key.
- Each file is encrypted independently — the sync service sees a forest of randomly-named blobs, not a container.
- Vault unlocking is managed by the Cryptomator app (menu-bar utility) which exposes a virtual mount via macFUSE.
- The cloud provider sees zero plaintext, even file names.

Cryptomator is complementary to FileVault: FileVault protects data at rest on the Mac's SSD; Cryptomator protects data in transit and at rest in cloud storage. They serve orthogonal threat models.

### 9. Forensics Acquisition: The Full Picture

| Scenario | What you can get | Notes |
|---|---|---|
| Live Mac, logged in, FileVault unlocked | Full logical + physical acquisition | Volume key is in SEP, SEP will answer decrypt requests. Standard imaging tools work (e.g., `dd` to external, Recon Imager, RECON ITR). |
| Live Mac, logged in, user asleep/screensaver | As above — key is still held | Sleep does not evict the VEK from SEP on Apple Silicon. Hibernation (`pmset -a hibernatemode 25`) does purge it. |
| Live Mac, at FileVault pre-boot screen | Nothing until authenticated | Key not yet released. Cannot bypass this screen without the password or PRK. |
| Powered-off Apple Silicon Mac | NAND is ciphertext | Cold acquisition yields encrypted blocks. No known bypass on M1/M2/M3/M4 era hardware without the credential. |
| Powered-off T2 Mac | NAND is ciphertext | Same as Apple Silicon for practical purposes; historical T2 exploits (checkm8-t2) required unpatched firmware. |
| Powered-off pre-T2 Intel Mac | If FileVault off: plaintext | If FileVault on, same encrypted-at-rest situation but software-keybag protection (weaker than SEP). |
| PRK or iCloud key available | Boot into recoveryOS, enter PRK at FileVault screen, then image live | Standard imaging workflow after recovery key unlocks |

**Live acquisition workflow (Apple Silicon, unlocked):**

```bash
# Identify the Data volume device node
diskutil list | grep -i data
# Typically /dev/disk3s5 or similar

# Image to external evidence drive (NVMe enclosure recommended)
sudo dd if=/dev/disk3s5 of=/Volumes/Evidence/mac_data.img bs=4m status=progress
# Or use rdisk (raw) for speed:
sudo dd if=/dev/rdisk3s5 of=/Volumes/Evidence/mac_data.img bs=4m status=progress
```

Note: On Apple Silicon, `/dev/disk3` is the synthesized "disk" view; actual NAND is on an internal `disk0` that Disk Utility exposes as a virtual block device. The `diskutil apfs list` command gives the authoritative volume map.

## Hands-on (CLI & GUI)

### fdesetup Reference

```bash
# Check current FileVault state
fdesetup status
# → FileVault is On. (or Off.)

# List users authorized to unlock at boot
sudo fdesetup list
# → user1,XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX
# → user2,YYYYYYYY-...

# Check if a personal recovery key exists
sudo fdesetup haspersonalrecoverykey
# → true (or false)

# Check if an institutional recovery key exists
sudo fdesetup hasinstitutionalrecoverykey
# → false (typical on personal machine)

# Enable FileVault (Tahoe: PRK not shown in stdout — use -outputplist)
sudo fdesetup enable -outputplist > ~/Desktop/fv-recovery.plist
# Immediately read and securely store the RecoveryKey value:
/usr/libexec/PlistBuddy -c "Print :RecoveryKey" ~/Desktop/fv-recovery.plist
# Delete the Desktop copy after storing safely
shred -u ~/Desktop/fv-recovery.plist  # or just rm; the file itself isn't sensitive
# after the key is extracted and stored

# Disable FileVault (requires admin password)
sudo fdesetup disable

# Add a user to the FileVault unlock list
sudo fdesetup add -usertoadd seconduser
# Prompts for both an existing enabled user's password and the new user's password

# Remove a user from the unlock list
sudo fdesetup remove -user seconduser

# Rotate the personal recovery key (generates a new one, invalidates old)
sudo fdesetup changerecovery -personal
# On Tahoe: use -outputplist for same reason as enable
sudo fdesetup changerecovery -personal -outputplist > ~/Desktop/new-recovery.plist

# Show detailed status including UUID
sudo fdesetup status -verbose

# List crypto users on a specific volume (more detail than fdesetup list)
diskutil apfs listcryptoUsers /
```

### Enabling FileVault via GUI

**System Settings → Privacy & Security → FileVault → Turn On...**

The GUI flow: enables FV, offers iCloud escrow (auto-selected if signed into iCloud) or local PRK display. On Tahoe this is the more reliable path for end-users who need to actually see their PRK before it disappears from screen.

### Creating an Encrypted APFS Volume

```bash
# Add an encrypted APFS volume to an existing container
# First identify the container reference:
diskutil list
# Look for APFS Container disk (e.g., disk3)

diskutil apfs addVolume disk3 APFS "SecureVault" -stdinpassphrase <<< "your-strong-passphrase"
# Or interactive:
diskutil apfs addVolume disk3 APFS "SecureVault" -passphrase
# Prompts for passphrase; auto-mounts at /Volumes/SecureVault
```

This volume is SEP-backed on Apple Silicon — passphrase-derived KEK wraps a SEP-managed VEK. Substantially stronger than an encrypted DMG.

### Creating an Encrypted Disk Image

```bash
# Create a 2 GB AES-256 encrypted sparse disk image
hdiutil create \
  -size 2g \
  -fs APFS \
  -encryption AES-256 \
  -volname "EncryptedVault" \
  -type SPARSE \
  ~/Documents/vault.sparseimage
# Prompts for passphrase twice

# Mount it (will prompt for passphrase, or use Keychain if stored)
hdiutil attach ~/Documents/vault.sparseimage

# Eject
hdiutil detach /Volumes/EncryptedVault

# Inspect image info (includes encryption method, size, format)
hdiutil imageinfo ~/Documents/vault.sparseimage | grep -E 'Encryption|Format|Sectors'
```

### Lab 2: Create and Exercise an Encrypted APFS Volume

> ⚠️ **Prep:** This creates a new APFS volume on your internal container. It is non-destructive but does consume a small amount of namespace. To roll back: `diskutil apfs deleteVolume /Volumes/LabVault` (or use Disk Utility → right-click the volume → Delete APFS Volume). Verify you have a current Time Machine or other backup before running against your boot drive.

```bash
# Step 1: Identify your APFS container
diskutil list internal | grep -E "Container|disk[0-9]"
# Note the Container disk identifier, e.g., disk3

# Step 2: Create an encrypted APFS volume (software-prompt passphrase)
CONTAINER_DISK="disk3"   # adjust to match your output
diskutil apfs addVolume $CONTAINER_DISK APFS "LabVault" -passphrase
# Enter a test passphrase when prompted. Note it — you'll need it.

# Step 3: Confirm it appears
diskutil apfs list | grep -A2 "LabVault"
ls /Volumes/LabVault

# Step 4: Write a test file
echo "encryption test $(date)" > /Volumes/LabVault/test.txt

# Step 5: Eject the volume
diskutil unmount /Volumes/LabVault

# Step 6: Remount — it will prompt for passphrase (not Keychain yet)
diskutil apfs unlockVolume $CONTAINER_DISK -mountpoint /Volumes/LabVault
# Enter your passphrase

# Step 7: Verify your test file survived
cat /Volumes/LabVault/test.txt

# Step 8: Inspect its crypto user record
diskutil apfs listcryptoUsers /Volumes/LabVault

# Cleanup when done:
diskutil unmount /Volumes/LabVault
diskutil apfs deleteVolume /Volumes/LabVault
```

### Lab 4: Rotate a Personal Recovery Key

> ⚠️ **DESTRUCTIVE to existing PRK.** After rotation, the old PRK is dead. If your new PRK is lost before you record it, and your password is also lost, you have no recovery path. **Before running this:** Ensure you have a working Time Machine backup. Ensure you know your login password. Have a text editor open and ready to paste the new key into your password manager immediately.

```bash
# Verify you have FileVault on before attempting
fdesetup status
# Should return: FileVault is On.

# Rotate the personal recovery key, capture via plist
sudo fdesetup changerecovery -personal -outputplist > /tmp/new-prk.plist

# Extract and display the new key immediately
/usr/libexec/PlistBuddy -c "Print :RecoveryKey" /tmp/new-prk.plist

# IMMEDIATELY copy this into your password manager / secure note

# Verify the key record changed (the Personal Recovery User UUID changes)
diskutil apfs listcryptoUsers /

# Securely delete the temp plist
rm /tmp/new-prk.plist

# Confirm the new PRK is registered
sudo fdesetup haspersonalrecoverykey
```

## Pitfalls & Gotchas

**PRK loss is permanent on Apple Silicon.** There is no SEP backdoor, no Apple support path, no data-recovery service that can help. The xART and UID binding mean the key hierarchy is hardware-specific and un-duplicatable. Treat your PRK with the same care as a seed phrase for a hardware wallet.

**FileVault "disabled" does not mean unencrypted.** Every Apple Silicon Mac has its Data volume encrypted at all times. Disabling FileVault just removes the pre-boot password gate. If you're advising a user that their data is "unprotected" because FileVault is off — you're wrong; you mean the boot is unprotected, not the data.

**Adding a user to FileVault is not the same as creating a macOS account.** `fdesetup add` only adds the user to the unlock list. The user must already have a local account. And if you create a new account _after_ FileVault was enabled, that account does NOT automatically get FileVault unlock rights — an admin must explicitly `fdesetup add -usertoadd newuser`.

**Hibernation and key eviction.** Normal sleep (`pmset -g | grep hibernatemode` returns 3) keeps the VEK in SEP during sleep. Safe Sleep / hibernatemode 25 writes a sleepimage to disk and the SEP evicts the key — the next wake requires full FileVault authentication. For high-security scenarios, `pmset -a hibernatemode 25` and `pmset -a standbydelay 0` enforce eviction on sleep. This also means cold-acquisition after an undisturbed sleep may face a locked screen requiring password.

**iCloud Recovery Key escrow is opt-in but default for iCloud users.** During `fdesetup enable` (GUI path) when signed into iCloud, Apple will offer to escrow. Accept consciously; don't let it slip through as a "next, next, done" UI. Corporate machines should use institutional keys, never iCloud escrow.

**Encrypted DMGs are NOT SEP-backed.** A `.dmg` or `.sparseimage` with AES-256 encryption is a software construction. The password goes through PBKDF2 derivation but the result lives in memory while mounted — unlike FileVault's VEK which never leaves the SEP. A memory dump of a Mac with a mounted encrypted DMG may yield the image key.

**`diskutil secureErase` still exists but is theatre on internal APFS.** The command works on external HFS+ drives where sector-level overwrite still matters. On internal Apple Silicon NVMe with APFS, issuing it on a FileVault-enabled volume accomplishes nothing that a CE didn't already do instantly. The UI has removed it from Disk Utility for the internal drive for this reason.

**Tahoe's `fdesetup enable` stdout change catches admins off guard.** Scripts that parsed stdout for the recovery key silently started capturing no key as of macOS 26. Always use `-outputplist` in automation.

## Key takeaways

1. Apple Silicon Data volumes are always encrypted — FileVault governs boot-time key release, not whether encryption happens at all.
2. The key hierarchy (UID → xART → VEK → KEK) means no single secret is sufficient without hardware; the UID fused into the SoC is irreplaceable.
3. Multiple KEKs can unlock one VEK — this is how multiple users, a recovery key, and an institutional key can all co-exist without re-encrypting data.
4. Cryptographic erase is instantaneous: destroying the VEK makes all NAND blocks permanently inaccessible, no overwrite required.
5. Cold acquisition of a powered-off Apple Silicon Mac without the password or PRK yields only ciphertext — there is no practical bypass.
6. Live acquisition (logged-in, unlocked machine) is the operative forensic path on Apple Silicon.
7. iCloud-escrowed PRKs are the lawful-access path Apple can respond to; for fully self-managed machines, Apple cannot assist.
8. On Tahoe (macOS 26), use `fdesetup enable -outputplist` or `changerecovery -personal -outputplist` to capture the PRK; stdout no longer shows it.

## Terms introduced

| Term | Definition |
|---|---|
| **VEK** | Volume Encryption Key — the AES-256 key that directly encrypts all blocks of an APFS Data volume |
| **KEK** | Key Encryption Key — a key that wraps the VEK, derived from user credential + hardware UID; multiple KEKs can protect one VEK |
| **SEP / Secure Enclave** | A co-processor in Apple Silicon (and T2) that manages cryptographic keys; keys never leave it |
| **UID** | Hardware UID — a 256-bit key fused into the SoC die; Apple does not have it; loss of hardware = loss of this key |
| **xART** | eXtended Anti-Replay Token — a hardware counter/nonce that prevents replay attacks on the key derivation |
| **PRK** | Personal Recovery Key — a 28-char Base32 KEK credential that can unlock the volume without the user password |
| **CE / Cryptographic Erase** | Destroying the VEK, rendering all encrypted data permanently inaccessible without any overwrite of data blocks |
| **Container Keybag** | APFS on-disk structure holding wrapped VEKs for all volumes in an APFS container |
| **Volume Keybag** | APFS on-disk structure in each volume holding the KEK records for all authorized credentials |
| **AES-256-XTS** | The mode of AES used for APFS block encryption (XEX-based Tweakable-codebook with ciphertext Stealing) |
| **SSV** | Signed System Volume — the cryptographically sealed read-only macOS System volume introduced in macOS 11 |
| **Sparsebundle** | An encrypted (or unencrypted) disk image format that stores data in fixed-size "band" files; used by Time Machine |
| **Cryptomator** | Open-source client-side encryption tool for cloud storage, encrypting each file independently before sync |
| **hibernatemode 25** | macOS power management mode that writes RAM to disk and evicts the VEK from SEP on sleep |
